# PRD 작성 프로토콜

`$do-next`에서 후보가 선택된 뒤 사용하는 호환/내부 프로토콜이다. PRD는 구현
지시서가 아니라 "왜 이 작업을 하는가, 무엇이 완료인가, 무엇은 하지 않는가"를
보존하는 제품 아티팩트다.

## 언어

PRD는 한국어로 작성한다. branch name, file path, command, TypeScript
identifier처럼 기계가 읽거나 코드와 직접 연결되는 표현만 영어를 유지한다.

## 입력

- 선택된 작업 후보
- 관련 raw PRD/ADR/notes
- `docs/wiki/index.md`의 현재 제품 방향
- 필요한 경우 현재 코드 구조

## 필수 섹션

```md
## 배경
## 목표
## 비목표
## 요구사항
### 기능 요구사항
### 비기능 요구사항
## 수용 기준
## 열린 질문
## ADR 필요 여부
## 관련 문서
```

## 작성 기준

- 배경은 사용자 또는 프로젝트 관점의 긴장을 설명한다.
- 목표는 이 작업이 달성해야 하는 결과를 적는다.
- 비목표는 범위 팽창을 막기 위해 구체적으로 적는다.
- 기능 요구사항은 사용자가 경험하거나 개발자가 구현해야 하는 동작을 적는다.
- 비기능 요구사항은 성능, 접근성, 반응형, 저장소 구조, 검증 가능성 같은 품질 조건을 적는다.
- 수용 기준은 검증자가 통과/실패를 판단할 수 있어야 한다.
- 열린 질문은 숨은 가정으로 묻지 말고 명시한다.
- ADR이 필요하면 어떤 결정을 내려야 하는지 적는다. 단, ADR 결정 자체는
  `proposed` 상태의 초안으로 남기고 사용자 승인 전 `accepted`로 바꾸지 않는다.

## 품질 게이트

PRD 초안은 아래 질문에 답할 수 있어야 한다.

- 왜 지금 이 작업인가?
- 이 작업이 끝났다고 어떻게 판단하는가?
- 어떤 일은 일부러 하지 않는가?
- 다음 작업 단위와 어떻게 연결되는가?
- 구현자가 설계 결정을 별도로 ADR에 남겨야 하는가?

## 실패 모드

- **나쁨:** "데이터를 만든다"처럼 선언만 있고 수용 기준이 없다.
- **좋음:** "필수 참조 데이터가 중복 없이 로드되고 누락 시 검증이 실패한다"처럼 관찰 가능한 기준을 둔다.

- **나쁨:** 구현 세부를 PRD에서 모두 확정한다.
- **좋음:** 제품 요구는 PRD에, 구조 선택은 ADR에 남긴다.

- **나쁨:** PRD 초안을 작성하자마자 `approved`로 둔다.
- **좋음:** 사용자 검토 전에는 `review`로 두고, 승인 시 `approval:` 근거와 함께 `approved`로 바꾼다.

- **나쁨:** 불확실한 부분을 임의로 확정한다.
- **좋음:** 열린 질문으로 남기고 다음 ADR/논의에서 해결한다.

## Handoff

새 작업에서는 `$do-next`가 아래 흐름을 오케스트레이션한다. 사용자가 PRD 방향과
필요한 ADR을 명시 승인하면:

```sh
npm run harness:start -- --type <type> --slug <slug> --title "<한국어 제목>"
```

생성된 raw artifact를 채운 뒤:

```sh
npm run harness:ingest -- docs/raw/<type>/<slug>
npm run harness:check
```

이 단계가 끝나도 구현은 시작하지 않는다. 구현 요청은 승인된 PRD/ADR을 기준으로
`feature-develop.md`로 넘어간다.
