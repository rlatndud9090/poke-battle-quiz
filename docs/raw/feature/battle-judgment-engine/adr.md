# ADR: 배틀 판정 엔진

## 컨텍스트

`pokemon-data-contract`가 main에 머지되어 정적 데이터(18타입 상성표 · 종족→가능특성 · ko/en 이름)는 확보됐다. 그러나 그것은 "사실"일 뿐, 게임의 핵심 루프인 **유저 행동 → 단서 → 타입·특성 추론**을 돌리는 규칙 계층이 없다. 유저가 "불 타입 물리 공격"을 했을 때 그것을 `x2`·`x0`·`방어-1·스피드+2` 같은 구조화된 단서로 바꿔주는 **판정 엔진**이 비어 있다. 이 모듈의 공개 계약은 이후 `guess-feedback-contract`(정답 은닉)·`battle-turn-ui`(배틀 UI)가 그대로 소비하므로, 시그니처·자료구조·확장 지점은 되돌리기 어려운 durable 결정이다.

- **PRD 요구:**
  - 유저의 한 행동(또는 정답 등장)을 **순수·결정론** 판정으로 구조화 단서로 변환(개념 연산 `judge`=행동 판정, `entryClues`=등장 판정).
  - **v1 = 4트랙**: (A) 공격[타입+물리/특수] · (B) 정답 등장 · (C) 상태이상기 · (D) 랭크변화기. 데이터셋 308종 전수 분석(`notes.md`)에 근거해 "유저가 자기 행동만으로(+정답 등장으로) 관측 가능한" 특성을 단서화한다.
  - **무효(x0) 단일 신호 통합**: 타입 면역(땅→비행)과 특성 면역(부유 등)을 유저가 구분 불가하게 한다.
  - v1 지원 특성을 분석 근거의 **명시적 화이트리스트**로 고정하고, 미지원 특성은 **거짓 단서 없이 안전 폴백**.
  - 특성 효과를 **데이터 기반 구조**로 분리해 신규 특성·신규 트랙(C5 "정답이 공격")을 **코어 판정 수정 없이** 붙일 수 있게 한다.
- **현재 코드/데이터 제약:**
  - `src/data`의 `Candidate`·`Ability`·`TypeChart`·`getEffectiveness`를 소비한다. **런타임 외부 fetch 0**(정적 import만), 의존은 `src/data → src/engine` **단방향**.
  - 데이터 계약의 `Multiplier`는 `0|0.5|1|2`이고 **단일 타입**용이다. 엔진은 복합 타입의 곱(`0.25/0.5/1/2/4`)과 C2 특성 보정으로 생기는 **비표준 배율**(예: thick-fat `0.5×`, filter류 `0.75×` 보정으로 `1.5×`/`3×`)까지 표현해야 하므로, 데이터 계약 `Multiplier`로는 표현 불가하다.
  - 데이터 계약엔 **스탯(능력치) 개념·상태이상·날씨/필드 식별자가 없다.** 엔진이 이 리터럴 타입들을 신규 정의해야 한다(아래 결정 (1)).
  - 표시명은 데이터 계약과 동일한 **고정 SHA PokéAPI api-data**의 `ability/<id>` `names[ko]`에서 결정론적으로 얻은 공식 한국어명을 권위 출처로 한다(임의 번역 금지).
- **사용자 경험 제약:**
  - **유저는 자기 포켓몬을 운용하지 않는다.** 추상 행동(공격/상태이상기/랭크변화기)만 고르고 정답을 대상으로 판정된다. "유저 측 스탯"이 없으므로, 원래 *시전자(상대)를 대상으로 되돌리는* 효과(intimidate 상대 공격↓, synchronize 상태 반사, mirror-armor 랭크하락 반사)는 델타로 표현할 대상이 없다 → **표식(marker)**으로만 노출한다.
  - **엔진은 "완전한 단서"를 생성한다. 정답 *식별*의 은닉은 엔진이 아니라 후속 `guess-feedback-contract`(전송 계층)의 책임이다.** 엔진 수준에서 단일 통합하는 것은 **x0(면역 출처) 단 하나** — 타입 면역 vs 특성 면역의 구분만 막는다. 랭크업·표식·상태결과·날씨 같은 그 외 관측 효과는 추론을 좁히는 *정상 단서*이며(형님의 "단서 풍부" 방향), 무엇을 유저에게 전송/표시할지는 은닉·UI 계층이 정한다.
- **검증 제약:** 순수함수이므로 Vitest로 트랙별·경계 케이스를 완결 검증한다(외부/전역 상태 없음, 동일 입력 동일 출력).
- **PRD 열린질문 해소 역참조:** PRD (P) 배율 도메인 → 본 ADR (2)·선택지 C. C1 흡수 부수효과 → 본 ADR (5)에서 "흡수랭크업 5종 노출 / 저수·축전·타오르는불꽃 순수 x0"로 해소.

## 결정

형님 승인 하에, 배틀 판정 엔진을 **순수·결정론·단발(턴 누적 없음)** 모듈 `src/engine/`로 두고 다음을 채택한다.

**(1) 두 순수 연산 + 판별 유니온 계약.** 엔진의 공개 표면은 두 순수 함수다(입력은 `Readonly`, 반환은 `readonly`로 불변성을 시그니처에 박는다).

- `judge(action: Readonly<Action>, secret: Readonly<Secret>): readonly Clue[]` — 유저 행동 판정.
- `entryClues(secret: Readonly<Secret>): readonly Clue[]` — 정답 등장 판정.
- `Secret` = 정답 후보(`candidate: Candidate`) + 그 후보의 가능 특성 중 **선택된 1개**(`ability: Ability`).

**신규 정의 리터럴 타입**(데이터 계약에 없어 엔진이 정의):

```ts
type StatId       = 'atk' | 'def' | 'spa' | 'spd' | 'spe'
type StatusId     = 'burn' | 'paralysis' | 'confusion' | 'poison' | 'sleep'
type WeatherId    = 'sun' | 'rain' | 'sandstorm' | 'snow' | 'harsh-sun' | 'heavy-rain'
type TerrainId    = 'grassy' | 'electric' | 'psychic' | 'misty'
type DamageMultiplier = number  // 정밀 곱(명목 별칭) — 데이터계약 Multiplier(0|0.5|1|2)와 별개
type EntryEffect =
  | { kind: 'weather';
      weather: WeatherId }
  | { kind: 'terrain';
      terrain: TerrainId }
  // 등장 시 정답 자신 스탯 상승(intrepid-sword 등)은 Clue.rank(target='secret')로 일원화(중복 표현 회피)
```

`Action`·`Clue` 판별 유니온(공통 판별자 `kind`; `Action.status`의 `StatusId`는 `Clue.status`와 동일 집합):

```ts
type Action =
  | { kind: 'attack'; attackType: PokemonType; category: 'physical' | 'special' }
  | { kind: 'status';
      status: StatusId }
  | { kind: 'stat'; stat: StatId; stages: number }   // v1 대표 = stat:'def', stages:-1

type Clue =
  | { kind: 'damage'; multiplier: DamageMultiplier }                       // 공격 배율(아래 (2))
  | { kind: 'rank'; target: 'secret'; stat: StatId; delta: number }        // 정답 자신 랭크 델타
  | { kind: 'status'; status: StatusId; result: 'applied' | 'immune' }     // 반사는 marker로
  | { kind: 'entry';
      effect: EntryEffect }
  | { kind: 'marker'; marker: 'intimidate' | 'status-reflect' | 'stat-reflect' }  // 시전자 재타깃
```

- `Clue.rank.target`은 v1 `'secret'` 고정. 대상 확장은 C5 피처가 판단하는 별개 seam이며, 단일 리터럴→유니온 확장은 소비처 재처리를 요구하는 breaking이라 v1엔 빈 확장 자리를 두지 않는다.
- `Clue.status.result`는 `applied`/`immune`만 — 반사는 `marker:'status-reflect'`로 일원화(별도 `reflected` variant 없음).
- 모든 소비처(렌더러·전송 계층)는 `kind` 망라 `switch` + `assertNever`로 누락을 컴파일 타임에 잡는다.

**트랙 → 산출 Clue kind 매핑**(다대다를 명시해 "4트랙 망라"를 검증 가능하게 한다):

| 트랙 | 입력 | 산출 가능한 `Clue.kind` |
| --- | --- | --- |
| (A) 공격 | `judge(attack)` | `damage` + `rank`(C1 흡수랭크업 + C3 피격 반응, 예: weak-armor) |
| (B) 등장 | `entryClues` | `entry`(날씨/필드 설치), `rank`(자신 부스트), `marker`(`'intimidate'`) |
| (C) 상태기 | `judge(status)` | `status`, `marker`(`'status-reflect'`) |
| (D) 랭크기 | `judge(stat)` | `rank`, `marker`(`'stat-reflect'`) |
| (C5, 후속) | — | `offense`(신규 kind, 아래 (4)) |

**막힘/무효 표현 규약.** "정답이 막았다"는 신호는 트랙마다 **의도적으로 트랙 고유 형태**다: 공격=`damage.multiplier:0`, 상태기=`status.result:'immune'`, 랭크기=`rank.delta:0`. 이를 한 형태로 통일하지 않는다(각 트랙의 자연스러운 0/무효 표현이 소비처 렌더링에 더 명확하고, 단일 추상화는 오히려 의미를 흐린다). 한편 **"조건 미스(반응 자체 없음)"**(예: weak-armor에 특수 공격)은 해당 `Clue`를 **배열에서 생략**한다. 즉 `delta:0`/`immune`(막힘, 명시적 단서)과 생략(단서 없음)은 구분된다.

**(2) 배율 = `DamageMultiplier`(= `number`, 데이터 계약 `Multiplier`와 별개 타입).** `damage.multiplier`는 정밀 곱이다. 타입 곱 × 특성 보정 곱이라 `0.125`·`1.5`·`3` 등 비표준값이 나올 수 있고, 데이터 계약 `Multiplier`(`0|0.5|1|2`, 단일 타입)와는 **별개**로 둔다. 명목 별칭 `DamageMultiplier`로 두어 `delta`·`stages` 같은 다른 `number`와 의도를 구분한다. 버킷화·표시 문구는 UI 몫이고, 부동소수 비교는 테스트에서 근사(epsilon) 허용한다. **x0는 단일 통합 신호**(`multiplier: 0`)로 타입 면역과 특성 면역을 구분 불가하게 낸다.

**(3) 특성 효과 표현 = 데이터 주도 화이트리스트 테이블 + 트랙별 해석기(하이브리드, 데이터 우선).** showdown 핸들러를 직접 포팅하지 않는다. 특성 효과를 **순수 데이터 레코드**(트랙 + 효과 파라미터 + `nameKo`)로 추출하고, **트랙당 해석기 1개**가 그 파라미터를 읽어 `Clue[]`를 생성한다. 특성이 수백 개여도 코드 분기는 트랙 수(약 4~6개)만큼이며, 데이터 순회로 역추론(특성↔단서)·스냅샷 테스트가 가능하다. 극소수 계산 의존 술어(filter류 "효과굉장 여부", multiscale "풀피", wonder-guard "효과굉장 아니면 x0" 등)는 트랙 해석기 내부의 **명명된 술어**(코드, escape hatch)로 격리한다.

**(4) C5 양방향 확장 seam = 트랙 레지스트리 단독.** 코어 오케스트레이터는 `tracks.flatMap(t => t.interpret(action, secret))` 형태이고, **C5(정답이 공격자)는 레지스트리에 새 트랙 1개를 추가**해 붙인다.
- *`direction` 파라미터는 두지 않는다.* 이 게임 모델엔 "유저 포켓몬"이 없어 기존 공격 트랙을 방향 반전해 재사용할 대상이 없다. C5는 "상성 단서의 방향 뒤집기"가 아니라 "정답의 공격 측 특성(위력 부스트·타입 변화)"이라는 **새로운 관측 종류**이므로 새 트랙이 맞다(YAGNI 회피).
- C5 트랙이 추가할 Clue kind의 **placeholder 스키마**: `{ kind: 'offense'; powerMod?: number; typeOverride?: PokemonType; ignoresDefensiveAbility?: boolean }`. 이는 seam이 추상 약속이 아니라 실제 부착 가능함을 보이기 위한 형태이고, **정확한 스키마·관측 가능성은 C5 피처가 확정**한다. (예: mold-breaker류 "방어특성 무시"는 방어자가 있어야 의미가 있어, 유저 포켓몬 없는 본 모델에서의 관측 가능성 자체를 C5 피처가 판단한다.)
- 새 트랙 추가 시 코어 `flatMap` 오케스트레이터와 기존 4트랙 해석기는 **무수정**이다.

**(5) 단발·통합·폴백·표식 정책.**
- **x0 통합**: 타입 면역과 특성 면역을 모두 `multiplier: 0`으로 통합. x0 단독으로는 둘을 구분 불가.
- **흡수+랭크업 노출(형님 결정)**: 피뢰침(전기무효+특공+1)·초식(풀무효+공격+1)·전기엔진(스피드+1)·마중물(특공+1)·노릇노릇바디(방어+2) **5종**(C1 12종 중 저수·축전·흙먹기=순수회복 x0, 타오르는불꽃=C5성격, dry-skin=C2 불약점, wind-rider=관측불가를 뺀 나머지)은 `multiplier: 0`과 **함께 `rank` 단서**를 내 "특성 면역"임이 드러난다. 단 저수(회복)·축전(회복)·타오르는불꽃(자기 불 강화=C5 성격)은 v1 관측 부수효과가 없어 **순수 x0**로만 낸다. `wind-rider`(바람타기)는 "바람 기술" move-flag 무효라 "타입+물리/특수" action 모델로 **관측 불가**(soundproof/bulletproof와 동일 논리) → v1 제외.
- **표식(marker)**: 위협(intimidate)·싱크로(synchronize 상태 반사)·미러아머(mirror-armor 랭크 반사)는 유저 포켓몬이 없어 스탯 델타가 아닌 표식으로 낸다. 이는 *의도된 단서*다(엔진은 완전한 단서를 내고, 어떤 단서를 전송·표시할지는 은닉/UI 계층이 결정).
- **안전 폴백 = "배율을 바꾸지 않는" 미지원 특성에 한해 타입 상성만**. 미지원이라도 *배율을 무효화/변경*하는 특성을 그냥 타입 상성으로 폴백하면 거짓 단서가 된다. 따라서 **배율을 바꾸는 면역/보정 특성은 전부 화이트리스트로 포섭**한다 — 폴백은 배율 비변경 특성에만 적용되어 거짓 단서가 구조적으로 불가능해진다.
  - `wonder-guard`(불가사의부적, 보유 1): "효과굉장(typeMod>0) 아니면 x0" 명명 술어 → 정확한 x0(폴백 던지지 않음).
  - `desolate-land`(끝의대지): "물 공격 x0" 술어. `primordial-sea`(시작의바다): "불 공격 x0" 술어. (둘 다 등장 트랙의 날씨 설치 단서도 동반.) — *복잡한 날씨 지속 시스템*은 비구현이되, 해당 타입 공격의 x0만 flat 술어로 정확히 낸다.
  - 반면 `wind-rider`처럼 action 모델로 애초에 트리거 불가한 특성은 폴백해도 거짓 단서가 없다(엔진이 x0를 주장하지 않고 유저가 "바람 기술"을 보낼 수도 없음) → 화이트리스트 밖으로 둔다.
- **단발**: 랭크는 누적하지 않고 그 행동의 델타만 반환한다. 랭크 트랙 해석기의 변형 적용 순서는 **차단(clear-body) > 반전·증폭(contrary/simple) > 반응(defiant/competitive)**으로 고정한다(contrary는 입력 `stages` 부호를 반전해 출력 `delta`에 반영).

**(6) 공식 ko명 자료화 위치 = 화이트리스트 데이터에 `nameKo` 칼럼 내장.** 별도 `ability-names.json`을 두지 않고, v1 화이트리스트 데이터 레코드에 `nameKo`를 함께 둔다. v1 단서에 등장하는 특성은 화이트리스트 부분집합(약 60~70종)이므로 308종 전체 명칭 테이블이 불필요하고, 효과 파라미터와 표시명 권위 출처가 한 레코드에 응집되어 정합이 깨질 여지가 없다. `nameKo` 값은 고정 SHA api-data `ability/<id>` `names[ko]`에서 결정론적으로 추출한다(표시는 UI 몫이되 출처 권위는 이 데이터). 전 특성 명칭이 필요해지면(C5 확장 후) 별도 파일을 재고한다.

## 선택지

### 선택지 A: 특성 효과 표현 — 순수 데이터 테이블 vs 핸들러 레지스트리 vs 하이브리드(채택)

특성을 단서로 바꾸는 표현 방식을 우리 클루 엔진 맥락(데이터 주도·단발·정답 역추론)에서 비교한다.

- **순수 데이터 테이블 + 카테고리 해석기**
  - 장점: 100% 직렬화·스냅샷 테스트·역추론 가능. 신규 특성=행 추가.
  - 단점: filter류("효과굉장이면 0.75×")·multiscale("풀피면 0.5×")·wonder-guard("효과굉장 아니면 x0")처럼 *계산 결과에 의존*하는 술어를 순수 JSON으로 표현 못 함 → 표현력 천장.
- **핸들러 레지스트리(특성 1개=함수 1개, showdown/pokerogue 방식)**
  - 장점: 표현력 무한.
  - 단점: 특성 수백 개 = 함수 수백 개로 showdown 회귀. 함수는 직렬화·정적 분석 불가, 역추론("이 단서셋으로 가능한 특성 후보는?")이 코드 역행을 요구해 정답 추론 목표에 정면 배치. 단발 엔진엔 과설계.
- **하이브리드 — 데이터 우선 테이블 + 트랙당 해석기 1개 + 명명된 술어 escape hatch (채택)**
  - 장점: 특성 ~95%가 선언적 데이터(역추론·확장·테스트 단순), 코드 분기는 트랙 수만큼만. 계산 의존 극소수만 **명명된** 술어로 격리해 메타로 인지 가능.
  - 단점: "100% 데이터"는 포기(술어 일부는 코드). 트랙 enum·효과 스키마의 초기 설계가 곧 천장이라 신중한 분류 필요.

### 선택지 B: C5 양방향 확장 seam — direction 파라미터 vs 트랙 레지스트리(채택)

C5("정답이 공격자")를 코어 수정 없이 붙일 확장 지점의 형태.

- **방향 파라미터(`direction` enum을 모든 트랙에 주입, 양방향 해석)**
  - 장점: 한 축으로 미래 양방향을 컴파일러가 강제. 이론상 우아.
  - 단점(기각 사유): 이 게임 모델엔 **유저 포켓몬이 없다.** 따라서 "정답이 유저를 공격"할 대상 타입/스탯이 없어 기존 공격 트랙을 *방향 반전*해 재사용할 v1 사례가 **0개**다. C5의 실제 사례(위력 부스트·타입 변화)는 방향 반전이 아니라 새 단서 종류라 어차피 레지스트리축으로 가야 한다 → `direction`은 v1에서 죽은 파라미터(YAGNI).
- **트랙 레지스트리 단독(채택)**
  - 장점: 새 단서 종류(C5의 `offense`)를 트랙 추가로 깔끔히 격리. 코어 `flatMap` 오케스트레이터·기존 4트랙 무수정. v1에 죽은 파라미터를 두지 않는다.
  - 단점: "기존 트랙을 방향만 반대로"라는 가상의 확장은 트랙 복제가 필요(그러나 본 게임 모델에 그런 요구가 없어 실해가 없다).

### 선택지 C: Clue 배율 타입 — 이산 집합 vs number(채택)

- **이산 화이트리스트 집합**(예: `0|0.25|0.5|0.75|1|1.5|2|3|4`)
  - 장점: 표시 버킷이 타입에 박혀 UI 단순, 비정상 값 타입 차단.
  - 단점: thick-fat×4배약점(→`0.125`) 등 곱 조합이 늘면 집합을 계속 확장해야 하고 누락 시 런타임 깨짐. 곱 결과를 매번 집합에 사상하는 비용·오차.
- **임의 `number`(별칭 `DamageMultiplier`, 채택)**
  - 장점: 타입 곱×특성 보정 곱을 정밀히 담아 확장 무제한. 데이터 계약 `Multiplier`와 별개 타입이라 단일타입 도메인 오염 없음.
  - 단점: "표시 가능 값인가"가 UI 버킷화로 이동, 부동소수 비교는 근사 허용 필요.

## 선택 근거

- **특성 표현은 하이브리드(A) 채택.** 1급 목표는 *정답 역추론*과 *데이터 주도 확장*이다. 핸들러 레지스트리는 함수를 역으로 못 돌려 역추론을 원천 차단하고 특성 수만큼 함수가 늘어 showdown으로 회귀하므로 기각. 순수 데이터 테이블은 역추론엔 최적이나 filter·multiscale·wonder-guard 같은 계산 의존 술어를 못 담아 천장에 부딪힌다. 하이브리드는 ~95%를 선언적 데이터로 두어 역추론·확장·스냅샷을 보존하면서 극소수 술어만 **명명**해 격리한다. (`notes.md`의 "showdown 핸들러를 직접 포팅하지 않고 카테고리별 산식 파라미터만 추출" 권고와 정합.)
- **C5 seam은 트랙 레지스트리 단독(B) 채택.** `direction` 파라미터는 우아해 보이나 유저 포켓몬이 없는 본 게임 모델에서 v1 사용처가 0이고 C5 실사례도 레지스트리축으로 가야 하므로 죽은 파라미터다 → YAGNI로 기각. 레지스트리 단독은 코어·기존 트랙 무수정으로 새 트랙만 추가하면 되고, C5 placeholder Clue kind까지 박아 seam을 구체화했다.
- **배율은 number(C) 채택.** 곱 조합이 `0.125`·`1.5`·`3` 등 비표준값을 낳아 이산 집합은 지속 확장·누락 위험을 안는다. `number`(별칭 `DamageMultiplier`)는 정밀·확장 무제한이며 데이터 계약 `Multiplier`와 별개 타입으로 단일타입 도메인 오염을 막는다. 표시 버킷은 UI 몫이라는 표현 분리 원칙과도 맞다.
- **ko명은 화이트리스트 내장((6)).** v1 단서 특성은 화이트리스트 부분집합이라 308종 전체 파일이 불필요. 효과 파라미터와 표시명 권위를 한 레코드에 응집하면 정합이 구조적으로 보장된다.

## 결과

### 긍정적 영향

- **소비자 계약 안정**: `judge`/`entryClues` + `Action`/`Clue` 판별 유니온이 `guess-feedback-contract`·`battle-turn-ui`가 의존할 안정 표면이 된다. 신규 리터럴 타입(`StatId`/`StatusId`/`WeatherId`/`TerrainId`/`EntryEffect`/`DamageMultiplier`)이 계약에 정의돼 소비처가 곧바로 소비 가능하고, `kind` 망라 `switch`로 누락이 컴파일 타임에 드러난다.
- **확장성(정직한 한정)**: *기존 트랙·기존 술어를 재사용하는* 신규 특성은 화이트리스트 데이터 **1행** 추가로 단서가 나오고 코어·해석기 시그니처는 불변. *새 계산 술어를 요구하는* 특성은 **명명된 술어(코드) 1개**가 동반되나, 이때도 트랙 해석기 시그니처와 코어 `flatMap` 오케스트레이터는 불변이다. C5 양방향은 레지스트리에 새 트랙만 추가해 붙는다.
- **검증성**: 순수·단발이라 Vitest로 트랙별·경계를 결정론적으로 덮고, 데이터 우선이라 화이트리스트 스냅샷 골든 테스트도 가능하다.
- **추론 게임 정합**: x0 통합(면역 출처 비노출)이 구조로 보장된다. 그 외 단서는 의도적으로 노출하고, *정답 식별*의 전송 은닉은 `guess-feedback-contract`가 별도로 책임진다(엔진=완전한 단서 생성, 은닉=전송 계층 — 책임 분리).

### 부정적 영향 / 트레이드오프

- **데이터 테이블 표현력 한계**: filter·multiscale·wonder-guard 등 계산 의존 술어는 순수 데이터로 못 담아 명명된 술어(코드)로 escape 한다. "100% 데이터"는 아니다.
- **x0 통합의 부분 완화**: 흡수+랭크업 노출(형님 결정)로 5종 특성 면역은 랭크 단서가 동반돼 *특성 면역임이 드러난다* → 그 특성들에 한해 정보 은닉이 부분 완화된다(의도된 트레이드오프). 저수/축전/타오르는불꽃은 순수 x0로 통합 유지.
- **marker/표식의 특성 시사**: marker(위협·상태반사·랭크반사)·흡수랭크업은 특정 특성을 좁히는 단서다. 이는 누수가 아니라 의도된 단서이며(정답의 *이름/ID*를 전송하지 않는 것이 은닉), 무엇을 전송·표시할지는 `guess-feedback-contract`·UI가 정한다. 본 ADR은 엔진이 **완전한 단서**를 생성함을 계약으로 한다.
- **number 배율의 표시 책임 이동**: "표시 가능 값인가"가 UI 버킷화로 넘어가고 부동소수 비교는 근사 허용이 필요하다.
- **트랙 스키마 초기 설계가 천장**: 트랙 enum·효과 파라미터를 잘못 그으면 확장 시 재설계 비용(showdown의 방어/공격 슬롯 분류를 출발 스키마로 차용해 위험 최소화).
- **C5는 여전히 비목표**: guts/marvel-scale 등 "상태 후 자기 공격 강화"는 v1 비관측(확장 시드만).
