# PRD: 배틀 판정 엔진

## 배경

`pokemon-data-contract`가 main에 머지되어 **정적 데이터**(18타입 상성표 · 종족별 가능 특성 · 이름)는 확보됐다. 그러나 이 데이터는 "사실"일 뿐, 게임의 핵심 루프인 **유저 행동 → 단서 → 타입·특성 추론**을 돌리는 규칙이 아직 없다. 유저가 "불 타입 물리 공격"을 했을 때 "x2"·"x0"·"방어-1·스피드+2" 같은 단서로 바꿔주는 **판정 규칙 계층**이 비어 있다.

이 엔진의 범위를 정하기 위해 데이터셋의 **고유 특성 308종을 전수 분석**했다(`notes.md`, Pokémon Showdown 핸들러 근거). 분석은 특성을 "이 게임에서 무엇으로 관측되는가"로 분류했고(C1~C8), 그 결과 v1 경계를 데이터 근거로 그을 수 있었다:

- **"유저가 공격만 해도" 관측되는 특성**(C1 면역/흡수·C2 배율 보정·C3 피격 시 랭크).
- **"정답이 등장할 때" 관측되는 특성**(C4 등장 발동 + C7 날씨/필드 "설치" 13종).
- **"유저가 상태이상기를 쓰면" 관측되는 특성**(상태 면역·반사 ~20종).
- **"유저가 랭크변화기를 쓰면" 관측되는 특성**(스탯하락 반응·차단·반사 14종).
- 반면 "정답이 공격해야" 보이는 특성(C5: 단단한발톱·우격다짐·타입 변화 등 52종/보유 497, 최대 버킷)은 구조적으로 다른 메커닉("정답도 행동")을 전제한다.

**전달 결정(형님 승인): v1은 네 행동/관측 트랙 — ① 유저 공격 판정 · ② 정답 등장 판정 · ③ 상태이상기 판정 · ④ 랭크변화기 판정 — 을 포함한다.** 즉 "유저가 자기 행동만으로(+정답 등장으로) 관측할 수 있는" 특성을 폭넓게 단서화한다. **오직 C5("정답이 공격해야 발동")만 v1 비목표**이되, 엔진 계약과 ADR은 C5(양방향 판정)까지 코어 수정 없이 붙일 **확장 시드**로 설계한다. 이 엔진의 공개 계약은 이후 정답 은닉 계약(`guess-feedback-contract`)과 배틀 UI(`battle-turn-ui`)가 그대로 소비한다.

> **게임 모델 가정(확정 컨셉 기반):** 유저는 자기 포켓몬을 운용하지 않는다. 유저는 **추상 행동**(공격 / 상태이상기 / 랭크변화기)만 고르고, 그 행동은 **정답 포켓몬을 대상**으로 판정된다. "유저 측 포켓몬 스탯"은 존재하지 않으므로, 랭크/스탯 단서는 **정답 자신**의 변화이거나 **관측된 효과 표식**이다. 원래 "상대(시전자)를 대상으로 되돌리는" 효과(intimidate의 상대 공격↓, mirror-armor·synchronize의 반사)는 대상이 없으므로 **표식**으로만 노출한다.

## 목표

- 유저의 한 행동(또는 정답의 등장)을 **구조화된 단서**로 바꾸는 **순수·결정론** 판정을 제공한다(개념 연산: `judge` = 행동 판정, `entryClues` = 등장 판정).
- **무효(x0)를 단일 신호로 통합**한다 — 타입 면역(땅→비행)과 특성 면역(부유 등)을 유저가 구분할 수 없게 한다.
- v1이 다루는 특성을 **데이터셋 분석에 근거한 명시적 화이트리스트**로 고정하고, 미지원 특성은 **거짓 단서 없이 안전 폴백**한다.
- 특성 효과를 **데이터 기반 구조**로 분리해, 신규 특성·신규 트랙(C5 "정답이 공격")을 **코어 판정 로직 수정 없이** 붙일 수 있게 한다.
- 기존 데이터 계약(`src/data`) 위에서 **런타임 외부 의존 0**으로 동작하고, **Vitest로 완결 검증**한다.

## 비목표

- **C5 "정답이 공격할 때 발동하는 특성"**(단단한발톱·돌머리·우격다짐·기술의달인·타입 변화 protean/libero/-ate 등 52종/보유 497) — v1 미구현. **확장 시드(붙일 자리)만 남긴다.**
- **"상태가 되면 자기 공격이 강해지는" 특성**(guts 근성·flare-boost·toxic-boost·quick-feet)과 marvel-scale — 정답이 공격해야 관측되는 C5 성격이라 제외. (상태이상기로는 "상태가 걸리나/막히나"까지만 관측하고, "상태 걸린 뒤 위력이 세지나"는 관측하지 않는다.)
- **C6 접촉/상태 응징 중 상태 면역·반사 관측 외**(접촉 반동 피해·도구 탈취·접촉 시 상태부여 등) 제외.
- **C7 날씨 수혜 특성**(쓱쓱·엽록소 등 ~18종)과 **날씨·필드의 지속 상태 시뮬레이션** 제외. v1 등장 단서는 "어떤 날씨/필드를 **설치**했나"만 노출한다.
- **날씨·필드의 지속 상태 시스템**: `desolate-land`/`primordial-sea`/`wonder-guard`의 *복잡한 지속 메커닉*(날씨 잠금, HP 의존 등)은 v1에서 구현하지 않는다. 단 **거짓 단서 방지를 위해 이들이 유발하는 x0는 flat 술어로 정확히 낸다**(wonder-guard=효과굉장 아니면 x0, primordial-sea=불 x0, desolate-land=물 x0; desolate/primordial은 날씨 설치 등장 단서도 동반). 즉 "안전 폴백"은 배율을 바꾸지 않는 미지원 특성에만 적용된다(ADR (5) 참조).
- **HP·실제 데미지 수치·급소·명중·기절·턴 누적·실제 배틀 진행** 제외. 랭크 변화는 누적하지 않고 **그 행동의 델타**만 반환한다.
- **정답 선택·정답 은닉·서버/Edge Function·UI·이름 자동완성·데일리 고정·결과 공유** 제외(후속 피처 소관).

## 요구사항

> PRD는 **"어떤 관측이 단서로 나와야 하는가"**까지 규정한다. 함수 시그니처·인자 순서·`Action`/`Clue` 자료구조·배율의 정확한 타입은 **ADR에서 확정**한다(아래 "ADR 필요 여부").

### 기능 요구사항

**(A) 공격 판정 트랙 (유저 행동 = 타입 + 물리/특수)**

- [ ] **데미지 배율 단서**: 단일 공격 타입이 정답의 (복합) 타입에 대해 갖는 배율을 단서로 낸다. 복합 타입은 곱이므로 결과 도메인은 `x0.25 / 0.5 / 1 / 2 / 4`를 **포함**한다(데이터 계약의 단일타입 `Multiplier`보다 넓다 — (P) 참조).
- [ ] **x0 통합**: 타입 면역(땅→비행)과 **C1 특성 면역**(부유·저수·타오르는불꽃 등)을 **구분 불가한 동일 x0 신호**로 낸다.
- [ ] **C2 방어 특성 보정**: thick-fat(불/얼음 0.5x)·heatproof(불 0.5x)·purifying-salt(고스트 0.5x) 등 화이트리스트 특성이 배율을 바꾸면, 그 보정이 반영된 배율을 낸다(타입만으론 설명 안 되는 배율 → 유저가 특성 추론).
- [ ] **C3 피격 반응 랭크 단서**: weak-armor(물리 피격 시 방어-1·스피드+2)·rattled(악/고스트/벌레 피격 시 스피드+1)·water-compaction(물 피격 시 방어+2) 등 **타입/분류 조건부** 화이트리스트 특성에 대해, **조건을 만족할 때만** 랭크 델타를 동반한다. 조건 미스 시 반응 없음.

**(B) 등장 판정 트랙 (정답 등장)**

- [ ] **정답 자신 스탯 등장 단서**: download·intrepid-sword(공격+1)·dauntless-shield(방어+1) 등 **정답 자신**의 스탯을 올리는 등장 특성을 정답 자신의 랭크 델타로 낸다.
- [ ] **날씨/필드 설치 단서**: C7 설치 13종(가뭄→쾌청, 잔비→비, 모래날림→모래바람, 눈퍼뜨리기→설경, 각 메이커→해당 필드 등)을 "설치된 날씨/필드 종류" 단서로 낸다.
- [ ] **표식형 등장 단서**: intimidate(위협)처럼 *원래는 상대를 대상으로 하는* 등장 특성은, 유저 측 포켓몬이 없으므로 스탯 델타가 아니라 **"위협 계열 등장 효과가 관측됨"이라는 표식**으로 낸다.

**(C) 상태이상기 판정 트랙 (유저 행동 = 대표 상태이상기)**

- [ ] **상태 결과 단서**: 유저가 대표 상태이상기(화상·마비·혼란·독·잠듦 중)를 쓰면, 정답에게 그 상태가 **적용됨 / 면역(걸리지 않음) / 반사(표식)** 중 어느 것인지 단서로 낸다. 면역은 보강2 화이트리스트(limber·water-veil·own-tempo·insomnia·immunity·purifying-salt·comatose 등 ~20종)로 판정하되, **어떤 특성이 막았는지 직접 노출하지 않고** "상태가 걸리지 않았다"는 관측만 준다(유저 추론).
- [ ] **반사 표식**: synchronize(싱크로)는 상태를 시전자에게 되돌리지만 유저 측 포켓몬이 없으므로 **"반사됨" 표식**으로 낸다(잠듦·얼음은 반사 대상 아님).

**(D) 랭크변화기 판정 트랙 (유저 행동 = 대표 랭크변화기)**

- [ ] **랭크 결과 단서**: 유저가 대표 랭크하락기(예: 방어-1)를 쓰면, 정답이 보이는 반응을 단서로 낸다 — **적용 / 차단(clear-body·white-smoke·hyper-cutter 등 → 델타 0) / 반응(defiant 공격+2·competitive 특공+2) / 반전(contrary) / 증폭(simple)**. 보강3 화이트리스트 14종 기준.
- [ ] **반사 표식**: mirror-armor는 받은 랭크하락을 시전자에게 되돌리지만 대상이 없으므로 **"반사됨" 표식**으로 낸다.

**(공통)**

- [ ] **Secret 모델**: 정답 = `Candidate`(데이터 계약) + 그 후보의 가능 특성 중 **선택된 1개**. 엔진은 이 단일 특성 기준으로 판정한다.
- [ ] **화이트리스트 + 안전 폴백**: v1 지원 특성은 `notes.md`(C1·C2·C3 + 등장 C4·C7-설치 + 보강2 상태면역 + 보강3 랭크반응)의 명시 목록으로 고정한다. 화이트리스트 밖 특성을 가진 정답은 **거짓 단서를 만들지 않고** 기본 판정(공격은 타입 상성만, 상태기/랭크기는 "정상 적용")으로 폴백한다.
- [ ] **확장 시드**: 특성 효과는 트랙/카테고리별 데이터·디스패치로 분리해, 신규 특성 1건 또는 신규 트랙(C5 양방향)을 **코어 판정 변경 없이** 추가할 수 있어야 한다.

### 비기능 요구사항

- [ ] **순수성·결정론**: I/O·전역 가변 상태·난수 없음. 동일 입력은 항상 동일 결과.
- [ ] **데이터 계약 재사용·단방향 의존**: `src/data`의 `Candidate`·`Ability`·`TypeChart`·`getEffectiveness`를 소비하고 **런타임 외부 fetch 0**(정적 import만). 의존은 `src/data → src/engine` **단방향**(data가 engine을 참조하지 않음).
- [ ] **특성 표시명 출처**: 단서에 특성·기술의 한국어명을 쓸 경우, 명칭은 데이터 계약과 동일한 **고정 SHA PokéAPI api-data**(`ability/<id>` 의 `names[ko]`)에서 결정론적으로 얻은 **공식 한국어명**을 따른다(임의 번역 금지). 표시명 자체는 UI 몫이되, 출처 권위는 이 계약을 따른다.
- [ ] **성능**: 단건 판정은 사전 구축된 화이트리스트/매핑 테이블 위에서 호출당 `O(1)~O(방어 타입 수)`로 동작한다(정답 후보 1209 × 특성 308 규모를 매 호출 순회하지 않는다).
- [ ] **표현 분리**: 엔진은 **구조화 데이터**만 반환한다. 한국어 문구·표시 형식·아이콘·"등장" 연출은 UI 계층(후속 피처) 몫이다.
- [ ] **검증성**: Vitest로 4트랙 대표 케이스 + 경계(0.25·4·x0 통합·C2 보정·C3 조건 매칭/미스·등장 설치·상태 면역/반사·랭크 차단/반응/반전·폴백)를 덮는다.
- [ ] **저장소 구조**: 엔진 코드는 데이터 계층과 분리된 도메인 모듈(예: `src/engine/`)에 둔다. 특성·기술 화이트리스트·효과 파라미터는 데이터로 분리한다.

## 수용 기준

**공격 트랙**
- [ ] **타입 배율**: 불→풀 = 2, 불→물 = 0.5, 불→(풀·벌레) = 4, 불→(물·바위) = 0.25.
- [ ] **x0 통합(핵심)**: 전기→땅(타입 면역)과 **부유 보유 정답에 땅 공격**(특성 면역)이 **동일한 x0 신호**를 반환해 유저가 둘을 구분할 수 없다. (단 C1 흡수+랭크업 5종(피뢰침·초식·전기엔진·마중물·노릇노릇바디)은 x0에 rank 단서가 동반되는 의도된 예외 — ADR (5) 형님 결정.)
- [ ] **C2 배율 보정**: thick-fat(두꺼운지방) 정답에 불 공격 → `0.5x`.
- [ ] **C3 피격 반응**: weak-armor(깨어진갑옷) 정답을 **물리** 공격 → `방어-1, 스피드+2`. **특수** 공격(분류 미스) → 반응 없음. rattled(주눅) 정답에 **노말** 공격(타입 미스) → 반응 없음.

**등장 트랙**
- [ ] **설치**: drought(가뭄) 정답 → `쾌청` 설치; grassy-surge(그래스메이커) 정답 → `그래스필드` 설치(날씨·필드 둘 다 동일 등장 단서 형식).
- [ ] **자신 스탯**: intrepid-sword(불요의검) 정답 → `정답 자신 공격+1`.
- [ ] **표식**: intimidate(위협) 정답 → `위협 계열 등장 효과 관측` 표식(특정 대상 스탯 델타 아님).

**상태이상기 트랙**
- [ ] **면역**: 전기자석파(마비) → limber(유연) 정답 = 마비 안 걸림. 도깨비불(화상) → water-veil(수의베일) 정답 = 화상 안 걸림.
- [ ] **반사 표식**: 전기자석파 → synchronize(싱크로) 정답 = `반사됨` 표식.
- [ ] **정상 적용**: 화이트리스트 밖 특성 정답 → 상태 정상 적용(거짓 면역 신호 없음).

**랭크변화기 트랙**
- [ ] **차단**: 꼬리흔들기(방어-1) → clear-body(클리어바디) 정답 = 델타 0.
- [ ] **반응**: 꼬리흔들기 → defiant(오기) 정답 = `공격+2`.
- [ ] **반전**: 꼬리흔들기 → contrary(심술꾸러기) 정답 = `방어+1`(하락이 상승으로).

**경계·공통**
- [ ] **C5 비관측 경계**: guts(근성) 정답에 도깨비불 → 화상은 **걸리되**(상태 적용), "공격이 세진다"는 단서는 **나오지 않는다**(C5, 비목표).
- [ ] **안전 폴백**: 화이트리스트 밖 특성을 가진 정답에는 거짓 단서 없이 기본 판정한다.
- [ ] **순수성**: 동일 입력 반복 호출이 동일 결과를 내고, 호출이 어떤 외부/전역 상태도 바꾸지 않는다.
- [ ] **확장성**: 신규 특성 1건을 데이터/디스패치 테이블에 추가하는 것만으로 해당 단서가 나오며, 코어 판정 함수는 수정되지 않는다.

## 열린 질문

- **(P) 엔진 출력 배율의 도메인**: 데이터 계약 `Multiplier`(`0|0.5|1|2`)는 **단일 타입**용이라 x4·x0.25, 그리고 C2 보정으로 생기는 **분수 배율**(filter/하드록의 "효과굉장 0.75x" → 2x가 1.5x로, 4x가 3x로)을 표현하지 못한다. v1 Clue 배율을 (a) 이산 화이트리스트 집합 / (b) 임의 number / (c) filter류 분수 보정 v1 제외 중 무엇으로 둘지 — **ADR 결정**.
- **상태이상기 대표 셋**: 화상·마비·혼란·독·잠듦 **5종 전부**를 v1 행동으로 열지, 일부(예: 화상·마비·혼란 3종)로 시작할지. 기술 표시명(도깨비불·전기자석파 등)은 공식명을 따른다.
- **랭크변화기 대표 셋**: 방어하락(꼬리흔들기) **1종**만 v1로 둘지, 공격하락·스피드하락 등도 포함할지(반응 특성 변별을 넓히려면 다종 필요).
- **등장 단서·표식 표현 형식**: 설치(날씨/필드)·자신 스탯·표식(위협/반사)을 어떤 공통 구조로 노출할지. 정답이 "등장"하는 게임 루프 표현은 `battle-turn-ui`(후보4)와의 계약에 의존.
- **C1 흡수 부수효과**: x0 통합 시 저수(회복)·피뢰침(특공↑) 흡수 부수효과는 **잠정 기본값 = 순수 x0**. 살릴지 여부는 ADR.
- **C5 확장 트랙 착수 시점**: "정답이 공격" 양방향 메커닉을 이 엔진 계약 직후 별도 피처로 열지, `battle-turn-ui` 이후로 둘지.

## ADR 필요 여부

**필요.** 다음은 되돌리기 어렵고 대안이 있는 durable 결정이며, 후속 피처가 소비할 모듈 경계다:

1. **엔진 공개 인터페이스** — `judge` / `entryClues` 시그니처와 4트랙을 아우르는 `Action`(공격/상태기/랭크기) · `Clue`(배율/랭크/상태/등장/표식) 판별 유니온의 모양, **그리고 Clue 배율의 타입·도메인**(데이터 계약 `Multiplier`와 **별개 타입**; 이산 vs number; filter류 분수 처리 — 위 (P)). `guess-feedback-contract`·`battle-turn-ui`가 이 계약을 그대로 소비한다.
2. **특성·기술 효과의 표현과 확장 시드** — 데이터 기반 테이블 + 트랙별 디스패치(하드코딩 핸들러 대비)와, **C5 "정답이 공격" 양방향**을 코어 수정 없이 붙일 확장 지점의 형태. 특성·기술 표시명의 **공식 한국어명 출처**(고정 SHA api-data) 계약 포함.
3. **단발 모델·통합·폴백 정책** — 순수 단발(확정) / x0 통합 위치 / 화이트리스트 미지원 시 폴백(공격=타입만, 상태기/랭크기=정상 적용) / C1 흡수 부수효과(잠정 순수 x0) / 등장·반사의 "표식 vs 스탯 델타" 표현 / wonder-guard·desolate-land 대량 x0 처리.

→ `$adr-helper`로 ADR(`proposed`)을 작성한다. 출발점은 위 3개 항목과 `notes.md`의 "ADR 결정 지점 후보"다.
