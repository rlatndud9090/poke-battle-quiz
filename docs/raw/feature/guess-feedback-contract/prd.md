# PRD: 추측/피드백 계약 및 정답 은닉 아키텍처

## 배경

배틀 판정 엔진(`src/engine`, `judge`/`entryClues` → `Clue`)이 머지되어 **단서 생성**은 됐다. 그러나 게임을 "성립"시키는 **데일리 루프** — 오늘의 정답 선정 · 세션(누적 단서·턴·추측) · 이름 추측 판정/피드백 — 이 비어 있다. 이 계층이 엔진의 `Clue`를 소비해 한 판의 게임을 만든다.

이 계층의 핵심 결정은 **정답 은닉을 서버로 빼느냐, 정적 SPA로 탈취를 감수하느냐**였다. 레퍼런스 조사(`notes.md` 「레퍼런스 조사」) 결론:

- 후보군이 작아(~1209) **정적 호스팅만으로는 진짜 은닉 불가** — 클라 해시/암호화/솔트는 전수 선계산(1209회)·번들 추출로 무력. **진짜 은닉은 서버(Edge Function)뿐.**
- 그러나 동형 인기작 **Squirdle은 순수 클라 판정·로컬 상태**로 운영된다(서버 없이 날짜 시드 결정론 정답) → 구조상 **탈취 감수형**. 솔로·무료·정적 지향에 풀서버는 과투자.

**형님 결정: Squirdle처럼 정적 SPA로 간다 — v1은 정답 탈취를 감수한다.** 진짜 은닉(서버/Edge Function/KV)은 v1 비목표.

단, 엔진 ADR이 못박은 **"엔진=완전한 단서 생성 / 정답 식별 노출 여부=전송 계층 책임"** 분리를 살려, 이 계층은 **얇은 Clue-only provider 경계**(로컬 구현)를 통과한다. v1 동작·UX는 Squirdle 그대로(정적·오프라인·탈취 감수)이되, 나중에 은닉이 필요해지면 **이 provider를 Edge Function 어댑터로 교체**하는 것만으로 승급할 수 있게 미래를 막지 않는다(비용 거의 0).

## 목표

- **데일리 게임 루프의 도메인(headless) 계층**을 제공한다: 날짜 시드 결정론 **정답 선정** + **세션 상태**(누적 단서·턴·추측 예산) + **행동→단서**(엔진 래핑) + **이름 추측 판정/피드백**.
- 정답 식별자를 **provider 인터페이스 수준에서 비노출**한다(해결 전): UI는 단서·결과만 받고, 정답의 candidateId/이름/abilityId는 응답·세션 직렬화에 담지 않는다. 로컬 구현이라 v1 실 은닉은 아니지만(탈취 감수), **인터페이스가 깨끗해 미래 서버 교체가 어댑터 수준**이 된다.
- **순수·결정론·정적**으로 동작한다(런타임 외부 fetch 0, 정적 호스팅), Vitest로 완결 검증.
- 데이터 계약(`src/data`)·엔진(`src/engine`)을 소비하고 **단방향 의존**을 지킨다.

## 비목표

- **진짜 은닉/서버/Edge Function/KV** — v1 미구현(정답 탈취 감수, Squirdle 선례). **단 Clue-only provider 시드는 남겨** 미래 승급을 어댑터 교체로 만든다.
- **UI·화면·입력 컨트롤·연출** — `battle-turn-ui`(후보4) 소관. 이 계층은 headless 도메인만.
- **인증·계정·리더보드·통계·공유** — 후속.
- **실 데이터 수집/생성** — 완료된 데이터 계약(`src/data`)을 소비만.
- **엔진 판정 규칙 변경** — 엔진은 그대로 소비(이 계층은 호출·누적·세션만).

## 요구사항

> 함수/타입의 정확한 시그니처·자료구조는 ADR(필요 시) 또는 구현에서 확정한다. PRD는 "어떤 동작·불변식이 필요한가"까지 규정한다.

### 기능 요구사항

- [ ] **데일리 정답 선정**: 날짜를 시드로 후보 풀(`src/data`의 전체 후보 — 모수는 데이터가 단일 출처, 현재 1209종)에서 **결정론적으로** 정답을 고른다. 후보(candidate) 선택 **그리고 그 후보의 `abilities` 중 한 특성 선택**까지 **모두 같은 시드로 결정**한다(엔진 `Secret`이 candidate + 선택된 ability를 요구 — ability 미선택은 결정론·비노출 양쪽을 무력화하므로 필수; 정확한 식은 구현 위임). 같은 날짜 → 같은 `(candidate, ability)`. **게임 날짜 경계 = UTC 자정 앵커**, **타임존 클램프 = `min(local, utc)`의 더 이른 게임 날짜**(쌍근식, 기기 시계 앞당김으로 미래 정답 선취 방지).
- [ ] **세션 상태(도메인)**: 누적 `Clue` 로그(턴 순서), **이동(move) 수**(행동·추측 통합 카운트), 게임 상태(`진행`/`해결`). 영속(새로고침 복원). **등장 단서는 시작 1회만 — 복원·재호출에도 중복 누적되지 않는 멱등.**
- [ ] **행동 제출**: 유저 행동(엔진 `Action`) → 엔진 `judge`/`entryClues` 호출 → `Clue[]`를 세션에 누적해 반환. 등장 단서(`entryClues`)는 게임 시작 시 1회.
- [ ] **추측 제출**: 추측 대상은 **candidate id**(정규화 완료된 식별자)로 받는다 — 이름→id 매핑·자동완성·동의어·폼 표기는 `battle-turn-ui` 소관(이 계층은 id 비교만). id가 정답 candidate와 일치하면 `해결` + 정답 공개, 아니면 `오답`(게임 계속). **이름 추측도 1 move로 카운트**(행동과 동일 화폐).
- [ ] **이동(move) 카운트·게임 종료(형님 결정)**: 추측은 **무제한**(횟수 제한·실패 상태 없음). **행동·이름 추측 모두 1 move로 통합 카운트**한다. 게임은 **정답 시에만 종료**하고, 종료 시 **정답까지의 move 수**를 점수로 기록한다(포케맨틀식 "몇 번 만에 맞췄나"). 많은 추측은 점수로 자연 페널티 → 브루트포스 자기억제(별도 제한 불필요).
- [ ] **Clue-only provider 경계**: 정답 `Secret`은 **provider 내부에 머문다** — provider가 엔진 호출(`judge`/`entryClues`)을 캡슐화해 `행동→Clue[]`·`추측→결과`만 노출하고, 정답 식별자(candidateId/nameKo/abilityId)는 **해결 전까지 provider 응답·세션 직렬화에 담지 않는다**(secret이 경계를 넘지 않음). 이 인터페이스는 로컬 구현 ↔ 서버 구현으로 **교체 가능**해야 한다. ⚠️ 이는 **인터페이스/직렬화 경계 한정**이며, v1 정적 번들·런타임 메모리에 정답 평문이 존재하는 것(탈취 가능)은 **비목표(탈취 감수)**다.

### 비기능 요구사항

- [ ] **순수성·결정론**: 날짜·행동 입력이 같으면 결과가 같다. 도메인 핵심은 I/O·전역 가변 상태 없이 순수, localStorage 접근만 얇은 영속 어댑터로 분리.
- [ ] **정적·외부 의존 0**: 런타임 외부 fetch 0(정적 import만). 정적 호스팅(예: Cloudflare/Vercel/GitHub Pages)으로 배포 가능.
- [ ] **의존 방향**: `src/data`·`src/engine`을 소비하고 역참조하지 않는다. UI 프레임워크 의존 최소화(테스트 가능한 headless 도메인).
- [ ] **교체 가능 경계**: 정답 provider·영속은 인터페이스 뒤에 둔다(**v1 기본 구현은 localStorage**). mock/로컬 ↔ 서버 어댑터 교체 시 도메인 코어 불변.
- [ ] **검증성**: Vitest로 데일리 결정론·타임존 경계·세션 누적·추측 판정·**move 카운트/점수**·**정답 비노출 불변식**을 덮는다.

## 수용 기준

- [ ] **데일리 결정론**: 같은 날짜 입력 → 같은 `(candidate, ability)`(특성까지 동일).
- [ ] **데일리 분포(균등·관찰가능)**: v1 정책은 **균등 추첨**. 전체 후보 수만큼의 연속 날짜를 돌리면 각 후보가 **충돌 없이 한 번씩**(전단사) 선택되거나, 표본에서 특정 후보 쏠림이 균등 PRNG 기대치를 벗어나지 않는다. (전자=순열/셔플 방식 / 후자=`seed mod N` 등 충돌 허용 방식 — 구현이 택한 모델에 따라 둘 중 하나로 검증. 인지도/난이도 가중·블랙리스트는 비목표.)
- [ ] **타임존 클램프**: 기기 시계를 다음날로 당겨도 게임 날짜는 `min(local, utc)`로 묶여 다음 정답을 선취하지 못한다. 정상 진행 시 **UTC 자정 앵커**로 다음 퍼즐 경계가 결정된다(경계 케이스 테스트).
- [ ] **행동→단서 누적**: 행동 제출 시 엔진 `Clue`가 세션 로그에 순서대로 쌓인다.
- [ ] **등장 단서 멱등**: 새로고침/복원 후에도 등장 단서가 재방출·중복 누적되지 않는다(시작 1회).
- [ ] **추측 판정**: 정답 candidate id 제출 → `해결` + 정답 공개 + move 수를 점수로 기록. 오답 → `오답`(게임 계속, move만 +1).
- [ ] **이동 카운트·점수**: 행동·추측 모두 move 수를 1 증가시킨다. 오답 추측은 게임을 끝내지 않고(무제한) move만 증가, 정답 시 `해결` + 그 move 수가 점수로 기록된다. `실패` 상태는 없다.
- [ ] **정답 비노출 불변식(인터페이스 경계)**: 게임이 `진행`인 동안 provider 반환 객체와 세션 직렬화 문자열을 검사하면 정답의 candidateId·nameKo·abilityId 문자열이 **부재**한다(해결 후에만 공개). ⚠️ 이는 인터페이스/직렬화 경계 검증이며 **번들·메모리 평문 부재는 보장하지 않는다**(탈취 감수 비목표).
- [ ] **영속 복원**: 새로고침 후 같은 날짜의 세션(누적 단서·move 수·상태)이 복원된다.
- [ ] **순수성**: 동일 입력 반복이 동일 결과를 내고 도메인 코어가 외부/전역 상태를 바꾸지 않는다.

## 열린 질문

- ✅ **추측 예산·턴 모델 (형님 결정, 해소)**: 추측 **무제한**, 행동·이름추측 통합 1-move 카운트, 정답까지의 move 수를 점수로 기록(포케맨틀식). `실패` 상태 없음.
- **provider 시드 채택 여부 → ADR 결정(형님)**: v1에 Clue-only provider 경계를 둘지(C-lite) vs 순수 인라인(pure Squirdle). 미래 은닉 승급 보존 vs 단순성 — `$adr-helper`에서 결정한다.
- ✅ **데일리 분포 정책 (형님 결정)**: 전체 후보 **균등 추첨**(전단사). 인지도/난이도 가중·블랙리스트는 비목표(필요 시 후속 선정 계층으로). ADR에서 시드 순열/셔플(전단사)로 확정.
- ✅ **게임 종료/재도전 (형님 결정)**: **하루 1판 고정(Squirdle식)** — `GameDate`당 1세션, 정답/포기 후 다음 날까지 대기. 연습·재도전 모드 없음(비목표).
- **localStorage 스키마/마이그레이션**: 날짜별 세션 키 구조, 정답 비노출 직렬화 형태.

## ADR 필요 여부

**필요(경량).** "정적 무은닉(Squirdle-like) + Clue-only provider 시드" 는 되돌리기 어려운 **아키텍처 결정**이다(은닉을 v1에서 포기한 근거, 상태·정답의 위치, provider 경계 모양, 미래 서버 승급 트리거). 레퍼런스 조사가 server/SPA/하이브리드를 비교한 결과를 결정으로 박아두면 추후 "왜 서버 안 썼나"가 추적된다.

- 채택 구조: 정적 SPA + 클라 데일리 결정론 + Clue-only provider(로컬) 경계.
- 선택지: A 풀서버/Edge · B 순수 정적(인라인) · **C 정적+provider 시드(채택 방향)**.
- → `$adr-helper`로 경량 ADR(`proposed`) 작성. 단순하면 PRD 결정으로 갈음 가능(형님 판단).
