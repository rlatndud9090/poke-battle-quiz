# ADR: 패널 플립 데일리 퀴즈 코어

## 컨텍스트

- **PRD 요구**
  - 하루 1회 같은 정답을 푸는 패널 플립형 데일리 퍼즐을 제공해야 한다.
  - 패널은 총 27개이며, 타입 18개 + 안다리걸기 + 비스트부스트 + 진화의휘석 + 메가스톤 + 다이맥스 + 타이틀 4개(`SwSh`, `LA`, `SV`, `ZA`)로 고정된다.
  - 패널 뒤집기와 이름 추측은 같은 예산을 쓰며, 첫 버전 총 예산은 12회다.
  - 12회를 모두 사용하고도 정답을 맞히지 못하면 실패 상태가 필요하다.
  - 타입 패널은 직관적으로 읽혀야 하므로, "지금 이 타입에 몇 배로 맞는가"를 단순한 `xN` 정보로 보여줘야 한다.
- **현재 코드/데이터 제약**
  - `src/data`는 현재 프로젝트의 정답 후보군으로 쓰이는 전체 `candidate universe`를 제공한다. 이 후보군은 원종뿐 아니라 타입/특성이 다른 일부 폼도 별도 candidate로 포함한다.
  - `src/session/dailyAnswer.ts`는 날짜별로 `candidate + selected ability`를 결정론적으로 고르는 로직을 이미 제공한다.
  - 현재 `src/session`은 배틀형 `clueLog` 세션 모델을 전제로 하며, 패널 공개 상태를 직접 표현하지 않는다.
  - 현재 런타임 데이터 계약에는 몸무게, 종족값, 진화 가능 여부, 거다이맥스 가능 여부, 타이틀 존재 여부가 없다.
  - 로컬 reference repo인 Pokémon Showdown 데이터에는 `weightkg`, `baseStats`, `evos`, `canGigantamax`, 세대별/작품별 `formats-data`가 있어 패널 메타데이터 생성 근거로 활용 가능하다.
- **사용자 경험 제약**
  - 사용자는 기존 배틀 시뮬레이션형 UX 대신 패널 공개 중심 루프를 원한다.
  - 첫 버전은 "전 포켓몬 지원 가능 시 그 범위를 유지"하는 쪽을 선호한다.
  - 10회는 전수조사상 최적 전략 기준 최소선에 가까워 보이므로, 첫 버전은 12회로 완충한다.
  - 이름만 맞히면 되는 경험을 유지하되, 현재 후보군이 폼을 포함하므로 "어디까지를 별도 정답으로 볼 것인가"를 명확히 해야 한다.
  - 메가진화 가능 여부를 패널로 제공하는 이상, 메가진화체 자체를 정답으로 넣으면 정보가 겹치거나 함정처럼 느껴질 수 있다.
  - 비스트부스트 패널 때문에 종족값 분배가 다른 폼이 실제로 구분 가능하더라도, 킬가르도 stance 같은 전투 중 상태 변화 폼이나 펌킨인 size 폼을 별도 정답으로 두는 것은 UX를 거칠게 만들 수 있다.
- **검증 제약**
  - 정적 SPA 원칙상 런타임 외부 fetch 없이 동작해야 한다.
  - 패널 메타데이터는 생성 단계에서 candidate 전체를 덮는지 자동 검증되어야 한다.
  - 실패/성공 결과, 패널 값, 날짜별 복원은 테스트 가능한 구조여야 한다.

## 결정

이번 기능은 아래 구조로 구현한다.

1. **정답 풀과 추측 성공 조건**
   - 첫 버전의 정답 풀은 현재 `src/data`의 전체 `candidate universe`를 **그대로 쓰지 않고**, 패널 게임용 `eligible answer pool`로 한 번 더 필터링해 사용한다.
   - 내부 정답은 기존처럼 `candidate + selected ability`다.
   - 추측 성공 조건은 **eligible answer pool 내부의 exact candidate 일치**로 둔다.
   - 포함 원칙:
     - 원종
     - 전투 시작 전에 고정되며, 플레이어가 별도 정답으로 납득할 수 있는 안정적 alternate form
     - 그 alternate form이 타입, 특성, 체중, 종족값, 작품 존재 여부 같은 패널 출력 중 하나 이상을 실제로 바꾸는 경우
   - 제외 원칙:
     - 메가진화, 원시회귀, 거다이맥스
     - 전투 중 일시적으로 바뀌는 상태 변화/변신 폼
     - 사이즈, 깃털색, 가족수처럼 "이름 맞히기"보다 "세부 subform 맞히기"에 가까운 외형 세분화 폼
   - 따라서 예시로는 지역폼, breed form, 전투 전 고정 타입/특성 변화 폼은 남길 수 있지만, 메가진화체, 킬가르도 stance, 펌킨인 size 분화 같은 후보는 첫 버전 정답풀에서 제외한다.
   - 사용자 입력은 "포켓몬 이름"으로 유지하되, 실제 제출 단위는 eligible candidate id이며, 표시 이름도 해당 candidate의 `nameKo`/`nameEn`을 사용한다.

2. **패널 메타데이터 공급 방식**
   - 타입 패널을 제외한 나머지 패널 값(안다리걸기, 비스트부스트, 진화의휘석, 메가스톤, 다이맥스, 타이틀 4종)은 **빌드 이전 생성 단계의 정적 패널 메타데이터 JSON**으로 공급한다.
   - 이 메타데이터는 현재 candidate 리스트를 기준으로 로컬 Pokémon Showdown reference repo를 읽어 생성한다.
   - candidate id와 Showdown species id 간 이름 차이가 있는 경우에는 명시적인 alias 매핑 테이블을 둔다.
   - 앱 런타임은 커밋된 정적 JSON만 읽고, reference repo나 네트워크를 직접 읽지 않는다.

3. **타입 패널 단순화 규칙**
   - 타입 패널은 "공격받았을 때의 최종 방어 상성 `xN`" 하나만 보여준다.
   - 계산에는 타입 상성과, **방어 상성을 직접 바꾸는 특성 효과만** 포함한다.
   - 포함 범위:
     - 타입 면역/흡수 계열로 인한 `x0`
     - 타입별 배율 증가/감소
     - `filter`/`solid-rock`/`prism-armor`처럼 효과 굉장할 때 배율을 줄이는 술어
     - `wonder-guard`, `desolate-land`, `primordial-sea`처럼 타입/효과굉장 여부만으로 결정 가능한 술어
   - 제외 범위:
     - 공격자 맥락이 필요한 효과
     - 물리/특수 분류 의존 효과
     - HP 조건 의존 효과
     - 피격 후 랭크 변화, 회복량, 반사 같은 부수 결과

4. **패널 게임 전용 세션 상태**
   - 기존 배틀형 `clueLog` 세션을 재사용하지 않고, **패널 게임 전용 세션 상태 모델**을 둔다.
   - 이 세션은 최소한 아래 정보를 저장한다.
     - `gameDate`
     - 공개한 패널 목록
     - 이름 추측 이력
     - 현재까지 사용한 예산(= 점수)
     - 게임 상태(`진행`/`해결`/`실패`)
   - 정답 reveal payload는 저장소에 직접 박제하지 않고, `gameDate`로부터 정답을 재유도해 파생 가능하게 유지한다.

5. **실패 시 공개 범위**
   - 실패 시에는 정답 이름만이 아니라, **정답 이름 + 선택 특성 + 정답 타입**까지 공개한다.
   - 이유는 12회 실패 후에도 플레이어가 “무슨 답이었는지”와 “왜 그런 패널 값이 나왔는지”를 학습할 수 있어야 하기 때문이다.

## 선택지

### 선택지 A: 기존 candidate universe를 그대로 재사용 + 정적 패널 메타데이터 + 전용 패널 세션

- 장점:
  - 이미 있는 데이터 계약과 데일리 정답 결정 구조를 최대한 재사용할 수 있다.
  - 전 포켓몬 지원에 가까운 현재 범위를 유지할 수 있다.
  - 패널 게임 상태를 기존 배틀형 세션에 억지로 끼워 넣지 않아 domain 경계가 깔끔해진다.
  - 메타데이터가 정적 JSON으로 고정되어 검증과 디버깅이 쉽다.
- 단점:
  - 메가진화체, 전투 중 상태 변화 폼, size 분화 폼까지 그대로 정답으로 들어와 UX 함정이 늘어난다.
  - Showdown alias 매핑 테이블 같은 연결 비용이 생긴다.
  - 패널 게임용 세션/메타데이터 생성 레이어를 새로 만들어야 한다.

### 선택지 B: 패널 게임용 eligible answer pool을 candidate universe에서 필터링 + 정적 패널 메타데이터 + 전용 패널 세션 (채택)

- 장점:
  - 기존 데이터 계약과 데일리 정답 결정 구조를 버리지 않고 재사용할 수 있다.
  - 지역폼처럼 플레이어가 별도 개체로 납득하는 안정적 폼은 살리면서, 메가/전투중 변화/size 분화를 정답 함정에서 뺄 수 있다.
  - 메가스톤 패널, 비스트부스트 패널과 충돌하는 UX를 줄인다.
  - 구현은 build-time filter 규칙과 검증으로 관리할 수 있어 추적 가능하다.
- 단점:
  - eligible filter 규칙과 예외 목록을 별도로 유지해야 한다.
  - "현재 candidate universe 전체"보다 정답풀이 줄어든다.
  - 어떤 폼을 남기고 뺄지 기준을 문서와 테스트로 꾸준히 유지해야 한다.

### 선택지 C: species 단위 정답 풀로 정규화 + 런타임 계산 중심 구조

- 장점:
  - 사용자가 체감하는 "한 마리" 개념은 더 단순해질 수 있다.
  - 정답 판정이 species 이름 기준이면 일부 폼 혼란을 줄일 수 있다.
- 단점:
  - 현재 candidate universe와 dailyAnswer 구조를 그대로 재사용할 수 없다.
  - 메가/지역폼/히스이폼처럼 패널 값이 달라지는 후보를 species 단위로 다시 합치는 정책이 추가로 필요하다.
  - 런타임에 필요한 패널 값 대부분이 현재 데이터 계약에 없어서 결국 별도 메타데이터가 다시 필요하다.
  - "이름만 맞히면 된다"를 species 단위로 완화하면, 현재 폼 포함 데이터 정책과 UX가 서로 어긋날 수 있다.

## 선택 근거

선택지 B를 채택한다.

- 사용자가 원한 것은 "복잡한 배틀 루프를 버리고 더 직관적인 퍼즐 루프로 전환"이지,
  현재 데이터 자산을 전부 갈아엎는 것이 아니다.
- species 단위로 전부 뭉개면 지역폼, breed form, 전투 전 고정 alternate form처럼 실제로
  패널 값이 달라지고 플레이어도 별도 정답으로 인식할 수 있는 경우까지 같이 사라진다.
- 반대로 현재 candidate universe를 그대로 쓰면 메가진화체, 전투 중 상태 변화 폼,
  size 분화 폼까지 exact candidate로 맞혀야 해서 UX 함정이 생긴다.
- 그래서 "현재 candidate universe를 source of truth로 삼되, 패널 게임용 eligible subset을 별도로 필터링"하는 중간 해법이 가장 현실적이다.
- 패널 값 대부분은 현재 런타임 계약에 없으므로, 어차피 생성 단계 메타데이터는 필요하다.
  그렇다면 런타임 계산을 억지로 늘리기보다 정적 JSON으로 고정하는 쪽이 정적 SPA 원칙과 맞는다.
- 타입 패널은 원래 사용자 의도부터 `xN` 정보 공개였고, 여기서 물리/특수, HP, 반사, 랭크 반응까지 넣으면
  다시 예전 배틀 중심 복잡성이 돌아온다. 따라서 "방어 상성을 직접 바꾸는 효과만" 반영하는 단순화 경계가 적절하다.
- 실패 시 이름+특성+타입을 공개하면 12회 제한으로 인한 좌절감을 줄이고, 다음 플레이를 위한 학습 루프를 만들 수 있다.

## 결과

이 ADR이 승인되면 새 컨셉 구현은 기존 `src/data`와 `dailyAnswer`를 토대로 하되,
그 위에 **정적 패널 메타데이터 레이어 + 전용 패널 세션 도메인 + 새 UI**를 얹는 구조로
진행한다. 기존 배틀형 UI 방향은 더 이상 기준 문서가 아니다.

### 긍정적 영향

- 새 컨셉의 핵심 루프가 데이터/세션/표시 구조까지 일관되게 정리된다.
- 구현자가 "정답 단위", "패널 값 출처", "실패 상태", "세션 shape"를 임의로 채우지 않아도 된다.
- 현재 데이터 자산을 재사용하면서도, 정답으로 냈을 때 납득하기 어려운 폼을 초기에 걷어낼 수 있다.
- 타입 패널의 의미가 사용자에게 훨씬 직관적으로 유지된다.

### 부정적 영향 / 트레이드오프

- eligible filter 규칙을 build-time에 유지보수해야 한다.
- 어떤 폼은 남고 어떤 폼은 빠지는지 문서와 테스트가 없으면 기준이 흐려질 수 있다.
- Showdown 기준 메타데이터 생성은 alias 유지보수 비용을 만든다.
- 기존 `src/session`과는 별도 domain이 생기므로 세션 관련 코드가 두 축으로 나뉜다.
- 실패 공개 범위를 넓히면 정답 closure는 좋아지지만, 스포일러 강도도 함께 높아진다.

## 후속 작업

- [ ] panel game용 eligible answer pool 규칙과 예시 목록을 코드/테스트로 고정한다.
- [ ] 패널 메타데이터 생성 스크립트와 산출물 스키마를 정의한다.
- [ ] candidate id ↔ Showdown id alias 목록을 정리하고 검증한다.
- [ ] 타입 패널 계산 로직에서 포함/제외 특성 목록을 코드로 고정한다.
- [ ] 패널 게임 전용 세션 타입과 localStorage 영속 구조를 정의한다.
- [ ] 실패/성공 reveal payload 형식을 패널 게임 도메인에 연결한다.

## 검증
