# PRD: 패널 플립 데일리 퀴즈 코어

## 배경

프로젝트의 기존 방향은 "배틀 행동을 고르고, 단순화된 판정 결과로 정답 포켓몬을
추론한다"는 배틀 시뮬레이션 중심 루프였다. 이 구조는 데이터·엔진·세션 같은
headless 토대를 쌓는 데는 유리했지만, 실제 플레이 경험 관점에서는 룰 이해 비용이
높고 한 판의 템포도 무겁다. 무엇을 누르면 어떤 정보가 나오는지 즉시 이해되지
않으면, 데일리 퍼즐로서의 진입 장벽이 커진다.

사용자 결정은 이 복잡도를 걷어내고, 핵심 경험을 **"패널을 뒤집어 정보를 점진적으로
사며 포켓몬을 맞히는 데일리 추론 게임"**으로 재정의하는 것이다. 즉, 플레이어는
배틀 행동의 상세 조합을 배우는 대신 **어떤 힌트를 언제 공개할지** 선택하고,
패널 뒤집기 수와 이름 추측 시도 수를 합친 낮은 점수를 목표로 플레이하되,
한 판에서 사용할 수 있는 총 행동 예산은 12회로 제한한다.

이미 저장소에는 전 세대 후보 데이터(`src/data`)와 날짜별 정답 결정론 시드
(`src/session/dailyAnswer.ts`)가 있다. 이번 작업은 이 자산을 버리지 않고, 새
컨셉에 맞는 **패널 메타데이터, 패널 게임 세션, 플레이어블 UI**를 얹어 실제 한 판을
완결할 수 있는 코어 루프를 만드는 작업이다.

## 목표

- [ ] 하루 1회 정해진 포켓몬을 맞히는 **패널 플립형 데일리 퍼즐**을 실제로 플레이할 수 있다.
- [ ] 사용자는 27개 패널 중 필요한 것만 뒤집어 정보를 수집하고, 임의 시점에 포켓몬 이름을 추측할 수 있다.
- [ ] 점수는 `뒤집은 패널 수 + 이름 추측 시도 수`로 일관되게 계산되며, 총 12회 예산 안에서 낮을수록 좋은 구조를 제공한다.
- [ ] 현재 저장소의 데일리 정답 결정 구조와 데이터 계약을 재사용하되, **패널 게임용 eligible 후보군**을 별도로 정의해 사용한다.
- [ ] 패널 값 계산, 점수 증가, 해결 판정, 영속 복원은 UI와 분리된 테스트 가능한 로직으로 제공된다.

## 비목표

- 기존 배틀 시뮬레이션형 UX: 행동 셀렉터, 배틀 로그, 상태이상/랭크 변화/등장 단서 중심 플레이
- 서버 기반 정답 은닉, 계정, 리더보드, 스트릭, 공유 이미지 생성, 통계 화면
- 패널 종류 추가 실험, 하드 모드, 점수 밸런싱 재튜닝
- 완전한 후보군 재큐레이션 또는 신규 데이터 소스 확장
- 스프라이트 연출, 정답 애니메이션, 소셜 최적화 같은 프레젠테이션 확장

## 요구사항

### 기능 요구사항

- [ ] **데일리 정답 유지**: 게임은 날짜별로 같은 정답을 사용해야 한다. 같은 날짜에는 같은 퍼즐을 보며, 날짜 경계 규칙은 기존 `GameDate` 정규화와 데일리 정답 결정 구조를 따른다.
- [ ] **시작 화면**: 게임 시작 시 닫힌 패널 그리드, 현재 점수, 날짜 정보, 추측 입력 영역을 함께 보여줘야 한다.
- [ ] **패널 그리드 구성**: 패널은 총 27개다.
  - 타입 패널 18개
  - 안다리걸기 패널 1개
  - 비스트부스트 패널 1개
  - 진화의휘석 패널 1개
  - 메가스톤 패널 1개
  - 다이맥스 패널 1개
  - 타이틀 패널 4개(`SwSh`, `LA`, `SV`, `ZA`)
- [ ] **패널 뒤집기 상호작용**: 닫힌 패널을 누르면 180도 회전하는 플립 전환 후 뒷면의 힌트 값이 보인다.
- [ ] **패널 영속성**: 같은 날짜 세션에서 한 번 공개한 패널은 새로고침 후에도 열린 상태를 유지한다.
- [ ] **공유 예산 12회**: 패널 뒤집기 1회와 이름 추측 1회는 모두 같은 행동 예산 1회를 사용하며, 한 판의 총 예산은 12회다.
- [ ] **점수 규칙**: 닫힌 패널을 처음 열 때만 점수 1이 증가한다. 이미 열린 패널을 다시 눌러도 점수는 증가하지 않는다.
- [ ] **이름 추측 규칙**: 포켓몬 이름 추측 제출 1회마다 점수 1이 증가한다.
- [ ] **게임 종료 규칙**: 오답 추측은 즉시 게임을 끝내지 않지만 남은 예산을 줄인다. 게임은 정답을 맞혔거나, 총 12회 예산을 모두 소진했을 때 종료된다.
- [ ] **실패 상태**: 12회를 모두 사용하고도 정답을 맞히지 못하면 실패 상태가 되며, 같은 날짜에는 추가 행동이나 재도전을 할 수 없다.
- [ ] **정답 공개**: 정답을 맞힌 뒤에는 정답 이름과, 내부 정답으로 사용된 선택 특성을 결과로 보여준다.
- [ ] **실패 공개**: 12회를 모두 사용해 실패했을 때는 정답 이름, 선택 특성, 정답 타입을 함께 공개한다.
- [ ] **패널 공개 순서 자유**: 사용자는 패널을 정해진 순서가 아니라 임의 순서로 열 수 있어야 하며, 패널을 하나도 열지 않고 바로 추측하는 것도 가능해야 한다. 단, 모든 선택은 총 12회 예산 안에서 이뤄진다.
- [ ] **타입 패널 값**: 각 타입 패널은 해당 타입 공격을 받았을 때의 최종 방어 상성을 `xN` 형태로 보여준다.
- [ ] **타입 패널 단순화 모델**: 타입 패널은 타입 상성과, 면역/흡수/배율 감소처럼 **방어 상성을 직접 바꾸는 특성 효과**를 반영해야 한다. 반면 공격자 맥락, HP 조건, 물리/특수 분류 의존 효과는 이번 컨셉의 타입 패널 범위에서 제외한다.
- [ ] **안다리걸기 패널 값**: 안다리걸기 패널은 타입 상성이나 실전 데미지 없이, 정답 포켓몬의 체중 구간에 따른 기본 위력만 표시한다.
- [ ] **비스트부스트 패널 값**: 비스트부스트 패널은 가장 높은 능력치를 `H/A/B/C/D/S` 체계로 표시한다. 최고 능력치가 둘 이상이면 `/`로 모두 표시한다.
- [ ] **진화의휘석 패널 값**: 진화의휘석 패널은 현재 정답 후보가 추가 진화를 할 수 있으면 `O`, 아니면 `X`를 표시한다.
- [ ] **메가스톤 패널 값**: 메가스톤 패널은 해당 정답 후보와 같은 종 라인에 메가진화가 존재하면 `O`, 아니면 `X`를 표시한다.
- [ ] **다이맥스 패널 값**: 다이맥스 패널은 해당 정답 후보와 같은 종 라인에 거다이맥스가 존재하면 `O`, 아니면 `X`를 표시한다.
- [ ] **타이틀 패널 값**: `SwSh`, `LA`, `SV`, `ZA` 패널은 각 타이틀에서 해당 정답 후보가 존재 가능한지 여부를 `O/X`로 표시하며, DLC 포함 기준을 따른다.
- [ ] **추측 입력 UX**: 사용자는 전체 지원 후보군에서 포켓몬 이름을 탐색 가능한 방식으로 추측할 수 있어야 한다. 입력 UX는 대규모 후보군을 실제로 사용할 수 있을 정도로 검색 가능해야 한다.
- [ ] **지원 후보군 범위**: 첫 버전은 현재 데이터 계약의 전체 candidate universe를 source로 삼되, 그중 **패널 게임용 eligible 후보군**만 정답 후보로 사용한다.
- [ ] **후보군 포함 원칙**: eligible 후보군에는 원종과, 전투 시작 전에 고정되며 플레이어가 별도 정답으로 납득할 수 있는 안정적 alternate form만 포함한다.
- [ ] **후보군 제외 원칙**: eligible 후보군에서는 메가진화/원시회귀/거다이맥스, 전투 중 상태 변화 폼, size·깃털색·가족수 같은 세부 subform 중심 후보를 제외한다.
- [ ] **세션 복원**: 같은 날짜에 다시 접속하면 공개한 패널 목록, 누적 점수, 추측 시도 이력, 해결 여부가 복원되어야 한다.

### 비기능 요구사항

- [ ] **정적 동작**: 런타임 외부 fetch 없이 정적 import만으로 동작해야 한다.
- [ ] **도메인 분리**: 패널 값 계산, 점수 규칙, 세션 전이는 React 컴포넌트 밖의 테스트 가능한 로직으로 분리한다.
- [ ] **패널 메타데이터 생성 가능성**: 안다리걸기, 최고 능력치, 진화 여부, 메가/거다이맥스, 타이틀 존재 여부 같은 패널 값은 빌드 이전 생성 단계에서 정적 산출물로 만들 수 있어야 한다.
- [ ] **데이터 검증성**: 패널 메타데이터가 누락되거나 candidate와 정합하지 않으면 검증 또는 테스트가 실패해야 한다.
- [ ] **반응형**: 모바일과 데스크톱 모두에서 패널 그리드와 추측 UI를 무리 없이 사용할 수 있어야 한다.
- [ ] **접근성**: 패널은 키보드로도 조작 가능해야 하고, 추측 입력은 대규모 후보군 탐색을 고려한 접근 가능한 입력 구조여야 한다.
- [ ] **확장성**: 이후 하드 모드, 공유 기능, 패널 추가/삭제 같은 후속 작업이 현재 코어 로직을 전면 재작성하지 않고 붙을 수 있어야 한다.

## 수용 기준

- [ ] 앱 실행 시 27개 패널이 렌더링되고, 닫힌 패널을 누르면 플립 후 값이 표시된다.
- [ ] 같은 패널을 두 번 눌러도 점수는 한 번만 증가한다.
- [ ] 패널을 3개 뒤집고 이름 추측을 2번 제출하면 점수가 5로 표시된다.
- [ ] 오답 추측 후 남은 예산이 있으면 게임은 계속 진행되며, 정답 추측 후에만 해결 상태와 결과가 표시된다.
- [ ] 패널 뒤집기와 이름 추측을 합쳐 12회를 모두 사용한 뒤에도 정답이 아니면 실패 상태가 표시되고, 추가 입력이 막힌다.
- [ ] 새로고침 후에도 열린 패널 목록, 점수, 추측 이력, 해결 여부가 같은 날짜 세션으로 복원된다.
- [ ] 타입 패널 계산은 대표 케이스에서 기대값을 반환한다. 예: 타입 면역, 특성 기반 면역, 특성 기반 배율 감소.
- [ ] 안다리걸기, 비스트부스트, 진화의휘석, 메가스톤, 다이맥스, 타이틀 패널 값이 대표 후보에서 기대값과 일치한다.
- [ ] eligible 후보군 전체에 대해 패널 메타데이터가 생성되며, 메타데이터 누락 candidate가 없다.
- [ ] 같은 날짜 입력에서는 같은 정답이 선택되고, 날짜가 바뀌면 새로운 퍼즐로 넘어간다.
- [ ] `npm run harness:gate` 통과에 필요한 검증이 구성된다.

## 열린 질문

- **패널 메타데이터 권위 출처**: 타이틀 존재 여부, 진화 가능 여부, 메가/거다이맥스 존재 여부를 어느 데이터 소스를 기준으로 생성할지 ADR에서 확정해야 한다.
- **타입 패널의 특성 반영 범위**: 어떤 방어 특성까지 "패널에서 보이는 정보"로 포함할지, 단순화 모델의 경계를 ADR에서 확정해야 한다.
- **후속 확장 모드**: 첫 버전에서 제외한 메가/전투중 상태 변화/size 중심 폼을 전문가 모드나 이벤트 모드에서 다시 다룰지 검토가 필요하다.

## ADR 필요 여부

**필요.** 이번 작업에는 되돌리기 어려운 구조 결정이 여럿 있다.

- 정답 풀과 추측 성공 조건을 existing candidate universe에서 어떤 eligible subset으로 연결할지
- 패널 메타데이터를 어떤 데이터 소스와 생성 파이프라인으로 공급할지
- 타입 패널의 단순화 규칙과 특성 반영 범위를 어디까지 둘지
- 패널 게임 세션 상태를 어떤 shape로 직렬화하고 복원할지
- 이름 추측 UX가 소비할 후보군 탐색 표면을 어떤 계약으로 둘지
