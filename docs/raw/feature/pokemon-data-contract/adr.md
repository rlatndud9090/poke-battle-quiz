# ADR: 포켓몬 데이터 계약(타입 상성표 · 종족→특성 매핑) 정의 및 전 세대 시드

전 세대 전종(~1025) 데이터를 어디서·언제·어떤 모양으로 확보해 정적 계약으로 제공할지에
대한 아키텍처 결정.

## 컨텍스트

PRD 요구: 전종(~1025)의 종족→가능특성(일반/숨은) 매핑 + 18타입 상성표 + 이름(ko/en)을
외부 공개 소스에서 자동 수집해 정적 산출물로 제공한다. 후속 엔진·UI·정답 판정이 모두 이
계약에 의존한다.

- **PRD 요구:** 전종 데이터 계약, 외부 자동수집, 빌드타임 정적 산출물, 결정론, 종 개수 정확값,
  gzip ≤ 50KB(notes 실측 시뮬레이션 ~18–26KB).
- **현재 코드/데이터 제약:** Vite SPA 정적 호스팅 지향, 추가 서버 인프라 없음. 아직 데이터·
  스키마 없음(컨셉 랜딩 수준).
- **사용자 경험 제약:** 데일리 퍼즐 → 정답 결정론·재현성 필요(외부 데이터 변동/장애에
  흔들리면 안 됨). 오프라인/안정성.
- **검증 제약:** 같은 소스 스냅샷 → 같은 산출물이 검증 가능해야 함. 타입표는 권위 출처와
  대조 가능해야 함. 종 모수는 고정 기대값으로 검증 가능해야 함.
- **외부 제약:** PokéAPI fair-use(라이브 대량 호출 비권장, 캐시 필수). ko 이름은 PokéAPI가
  사실상 유일한 공개 자동 소스(Showdown/Smogon/veekun 미제공). 포켓몬 IP는
  Nintendo/Game Freak/TPC — 비공식 팬·비상업.

(상세 조사 결론과 출처는 `./notes.md`)

## 결정

1. **수집 소스:** PokéAPI를 1차 소스로 하되, 라이브 호출이 아니라 `PokeAPI/api-data`
   GitHub 정적 덤프에서 수집한다.
2. **생성 시점:** 빌드타임 수집 스크립트가 정제(trim)·검증해 정적 JSON 산출물을 저장소에
   커밋한다. 런타임은 정적 import만 하고 외부 fetch는 하지 않는다.
3. **소스 리비전 고정:** 수집 스크립트는 `api-data`의 **커밋 SHA**를 고정 참조하고, 그 SHA와
   덤프 레포 URL을 산출물 메타데이터에 기록한다(태그가 아니라 불변 SHA를 채택).
4. **타입 상성표 권위 출처:** PokéAPI `type.damage_relations`를 1차 출처로 18×18로 펼쳐
   채택하고 Showdown typechart로 2차 검증한다. 18타입만 화이트리스트(stellar/unknown 제외).
5. **세대(generation) 표현:** species 데이터에 **명시 필드로 저장**한다(도감번호 파생 아님).
6. **한국어 이름:** ko는 PokéAPI species `names`에 단일 종속한다. ko 누락 종은 en으로
   폴백하고 누락 목록을 산출물 메타데이터에 기록한다.
7. **종 모수·폼 정책:** 메가진화/거대화/지역폼은 `baseSpecies`/`forme`로 필터해 **본종만**
   채택한다(지역폼을 별도 정답 후보로 둘지는 후속 결정). 비배틀 타입(stellar/unknown)은
   제외한다. 최종 종 모수는 스냅샷 기준 **1025 − 제외수**이며, 제외 목록·사유를 산출물
   메타데이터에 기록한다.

## 선택지

### 축 1 — 수집·생성 방식

#### 선택지 A: 빌드타임 수집(api-data 덤프) + 정적 커밋 — 채택

- 장점: 정적 호스팅에 완벽(추가 인프라 0), 런타임 외부 의존 0, fair-use 위반 없음(수집은
  빌드 1회), 데일리 정답 결정론·재현성, 오프라인/안정. 초기 로드 gzip ~수십 KB 단일 번들.
  api-data 덤프는 ko 포함 + 라이브와 동일 구조라 trim만 하면 됨.
- 단점: 신세대 출시 시 재수집·재커밋 필요(수년 주기로 드묾). 산출물이 저장소 용량을
  소량 차지.

#### 선택지 B: 런타임 PokéAPI fetch

- 장점: 항상 최신 데이터.
- 단점: 외부 장애 시 앱 마비, 사용자마다 호출 → fair-use/IP 밴 리스크, 전종 2050+ 호출 또는
  복잡한 캐시, 데일리 정답이 외부 변동에 흔들림, 정적 호스팅 이점 상실.

#### 선택지 C: 빌드타임 + 라이브 PokéAPI 대량 호출

- 장점: 덤프 레포 없이 최신 수집.
- 단점: 수집 시 2050+ 라이브 호출 = fair-use 위반 소지, 네트워크 불안정에 빌드가 취약.

### 축 2 — 세대(generation) 표현

#### 선택지 A: species에 generation 명시 필드 저장 — 채택

- 장점: 추론 힌트 축(세대)으로 즉시 사용 가능, 폼/지역폼/특수 도감번호 예외에 흔들리지
  않음, 스키마가 명확하고 안정적.
- 단점: 종당 정수 1필드 추가(번들 비용 미미), 수집 시 species→generation 매핑 단계 필요.

#### 선택지 B: 도감번호(num) 범위에서 파생

- 장점: 저장 0, Showdown `dex-species.ts` 패턴.
- 단점: 파생 규칙이 폼/지역폼/특수번호에서 깨질 수 있음, 힌트 사용 시 매번 계산,
  규칙이 코드에 흩어져 스키마 안정성(PRD 비기능요구)과 충돌 소지.

### 축 3 — 소스 리비전 고정(pin) 방식

#### 선택지 A: 커밋 SHA 고정 — 채택

- 장점: 불변·정확. 덤프 레포 내용이 갱신돼도 동일 스냅샷을 영구 보장 → 결정론 전제 충족.
- 단점: 가독성 낮음(40자 해시). 메타데이터에 함께 기록해 보완.

#### 선택지 B: 태그/릴리스 고정

- 장점: 가독성, 의미 있는 버전.
- 단점: 태그가 이동/삭제될 수 있어 불변성이 약함. `api-data`는 정식 릴리스 태그 체계가
  약해 신뢰할 고정점이 못 됨.

### 축 4 — 타입 상성표 1차 권위 출처

#### 선택지 A: PokéAPI `damage_relations` — 채택(1차)

- 장점: 기계 가독(JSON), ko 소스와 동일 레포(api-data)라 파이프라인 단일화, Bulbapedia와
  교차검증됨.
- 단점: 18타입 외 항목(stellar/unknown) 필터 필요(경미).

#### 선택지 B: Showdown `typechart`

- 장점: 정수코드(0/1/2/3)로 간결, 2차 검증에 적합.
- 단점: 1차로 쓰면 `damageTaken` 인코딩 변환 + "방어자 기준" 구조 변환 필요, ko 소스와
  레포가 분리됨. → **2차 검증용으로만** 사용.

#### 선택지 C: Bulbapedia

- 장점: 권위가 높음(커뮤니티 표준).
- 단점: 위키라 자동수집에 부적합(HTML 파싱 취약·불안정). → 교차검증 참고로만.

## 선택 근거

- **수집·생성은 축1-A 채택:** 데일리 퍼즐의 핵심인 정답 결정론·재현성과 정적 호스팅
  적합성을 동시에 만족하는 유일안이다. B(런타임)는 외부 변동·장애·fair-use 때문에 데일리
  게임에 부적합하고, C(라이브 대량 호출)는 fair-use 위반 소지가 크다. api-data 덤프는 ko를
  포함하고 라이브와 동일 구조라 정제 비용이 낮다.
- **세대는 축2-A(명시 저장) 채택:** 추론 힌트로 쓸 가능성과 파생 규칙의 예외 취약성 회피,
  스키마 안정성을 위해서다. 번들 비용이 무시 가능해 트레이드오프가 작다. B(파생)는
  폼/특수번호에서 깨지기 쉬워 기각.
- **pin은 축3-A(커밋 SHA) 채택:** "같은 스냅샷 → 같은 산출물"이라는 결정론은 고정점이
  불변일 때만 성립한다. 태그(B)는 이동 가능해 불변성이 약하므로 기각. SHA + 레포 URL을
  메타데이터에 기록해 가독성을 보완한다.
- **타입표는 축4-A(PokéAPI) 채택:** Bulbapedia와 교차검증돼 권위가 충분하고 ko 소스와 동일
  레포라 파이프라인이 단일화된다. Showdown(B)은 인코딩/구조 변환이 필요해 1차로는 비용이
  크고 2차 검증에 더 적합하다. Bulbapedia(C)는 자동수집에 부적합해 참고용으로만 쓴다.
- **ko 단일 종속은 대안 부재:** 타 공개 소스가 ko를 제공하지 않으므로 PokéAPI에 의존하되,
  en 폴백과 누락 기록으로 리스크를 관리한다.
- **종 모수는 본종 우선:** 메가/거대화/지역폼을 별도 엔트리로 두면 모수·정답 후보가 모호해진다.
  baseSpecies 필터로 본종만 채택해 모수를 1025−제외수로 고정하고, 폼 포함은 후속에서 결정한다.

## 결과

### 긍정적 영향

- 후속 엔진·UI·정답 판정이 안정적인 정적 계약 위에서 작동한다.
- 외부 의존·장애·rate-limit과 무관한 결정론적 데일리 게임 토대를 확보한다(고정 SHA 스냅샷).
- 신세대 확장이 "SHA pin 업데이트 + 재수집 PR"로 국소화된다.

### 부정적 영향 / 트레이드오프

- 데이터 신선도가 수동 재수집에 의존한다(신세대는 수년 주기라 수용 가능).
- 산출물(JSON)이 저장소에 커밋된다(gzip ~18–26KB 실측, 예산 ≤50KB 이내라 수용 가능).
- ko 이름 품질이 PokéAPI에 종속된다(누락 시 en 폴백).
- 폼(메가/거대화/지역폼)을 본종으로 접어 정답 후보에서 제외하므로, 폼 기반 추론은 후속
  결정 전까지 불가하다.

## 후속 작업

- [ ] 수집 스크립트 구현(api-data 커밋 SHA pin, trim, 스키마 검증, 제외 목록·메타데이터
      기록) — 본 피처 구현 단계.
- [ ] 타입 정의·정적 산출물·조회 헬퍼 구현 — 본 피처 구현 단계.
- [ ] 특성 식별자 → 효과 매핑, 상태이상·랭크 변화 규칙은 `battle-judgment-engine` 피처에서
      이 계약의 식별자에 연결(본 ADR 범위 밖).
- [ ] 지역폼 별도 정답 후보 포함 여부 — 후속 결정(본 ADR은 본종 우선으로 고정).

## 검증

- [ ] 수집 산출물이 고정 SHA 스냅샷에서 결정론적으로 재생성된다(같은 입력 → 같은 출력).
- [ ] **종 개수가 스냅샷 고정 기대값(1025 − 제외수)과 일치한다**(메타데이터의 제외 목록과 대조).
- [ ] 18×18 상성표가 PokéAPI `damage_relations` 파생 기대 스냅샷과 전체 일치한다(vitest).
- [ ] 전종 스키마 검증(타입 1~2개, 특성 1개 이상, generation, ko/en 비어있지 않음)이 통과한다.
- [ ] 데이터 산출물 gzip 합계가 예산(≤50KB) 이내다.
- [ ] `npm run harness:gate`가 통과한다.
