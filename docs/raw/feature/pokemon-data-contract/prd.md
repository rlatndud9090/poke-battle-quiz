# PRD: 포켓몬 데이터 계약(타입 상성표 · 종족→특성 매핑) 정의 및 전 세대 시드

## 배경

poke-battle-quiz는 유저가 매 턴 '배틀 행동'을 하고 단순화 배틀 엔진의 판정(데미지
배율 · `x0` 무효 · 능력 랭크 변화)을 단서로 정답 포켓몬을 추론하는 데일리 퍼즐이다.
이 게임의 세 축 — 배틀 판정 엔진, 추측/피드백 계약, 플레이 UI — 은 모두 "포켓몬을
어떤 데이터 모양으로 표현하는가"에 의존한다. 그런데 지금은 그 **공유 데이터 스키마가
없다.** 스키마가 없으면 엔진은 타입 상성을, UI는 후보 이름군을, 정답 판정은 종족·특성
풀을 각자 다른 모양으로 가정하게 되고, 이후 모든 작업이 서로 어긋난다.

따라서 첫 피처로 **타입 상성표 + 종족/폼→가능특성 매핑 + 이름**이라는 데이터 계약을
정의한다. 형님 결정에 따라 대상은 1세대 한정이 아니라 **현존 전 세대 모든 종(~1025)**
이며, 데이터는 손으로 큐레이션하지 않고 **외부 공개 소스(PokéAPI 등)에서 자동 수집**한다.
또한 이 게임은 타입·특성으로 정답을 좁히므로, **타입 또는 특성이 원종과 다른 폼(지역폼·
메가진화·원시회귀)은 별도 정답 후보**로 포함한다(외형·스탯만 다른 폼은 제외). 이로써
데이터의 출처·수집 시점·폼 정책·스키마가 durable decision이 되며, 순수 데이터/타입이라
서버 없이 한 브랜치에서 끝나고 검증도 명확하다.

## 목표

- [ ] 18타입을 식별하는 타입 정의와 타입 상성표(공격×방어, 배수 0/0.5/1/2)를 **단일
      출처**로 제공하고, 알려진 상성을 정확히 반영한다.
- [ ] 현존 전 세대 전종(~1025) **및 타입/특성이 원종과 다른 폼**의 정답 후보 데이터
      (종족/폼→가능특성(일반+숨은 구분) 매핑 · 타입 · 세대 · 이름(ko/en))를 TypeScript
      타입으로 정의하고 데이터가 그 타입을 만족하게 한다.
- [ ] 데이터를 **외부 공개 소스에서 빌드타임에 자동 수집·정제·검증**해 정적 산출물로
      만들고, **수집 소스 리비전을 고정(pin)** 해 재현 가능하게 한다(런타임 외부 fetch 없음).
- [ ] 도메인 코드가 쓰는 **조회 헬퍼**(후보 조회, 타입 상성 조회)를 제공해 후속 엔진·UI가
      같은 계약 위에서 작동하게 한다.

## 비목표

- 데미지 배율 **계산/판정 로직**(→ `battle-judgment-engine` 피처).
- 능력 랭크 변화(랭크업/다운) · **상태이상** · 전투 시뮬레이션 · HP/실데미지/턴 누적 등
  배틀 진행 규칙(→ `battle-judgment-engine` 피처).
- 어떤 **UI 렌더링**(→ `battle-turn-ui` 피처).
- 정답 선택 · 정답 은닉 · 서버/Edge Function, 일일 정답의 **같은 종 폼 그룹핑**(→ 후속 피처).
- 특성의 **효과·발동 시점·작동 규칙**(→ `battle-judgment-engine` 피처). 특성의 효과는
  배틀 엔진 로직에 강결합이므로, 이 계약은 특성을 **식별자(이름) + 종족별 일반/숨은 슬롯**
  으로만 기록하고 효과·발동 메타데이터는 두지 않는다(효과 매핑은 후속 엔진 피처가 이
  식별자에 연결한다).
- **거대화(gmax) 및 타입·특성이 원종과 동일한 폼**(외형·스탯·코스튬, 예: deoxys 폼,
  피카츄 모자) — 추론 단서가 원종과 같아 별도 후보로 두지 않는다.
- **아르세우스·실버디 종 전체** — 타입만 18종 룰렛이라 추론 게임에서 노이즈이므로 노말
  원종까지 블랙리스트로 제외한다.
- 스탯 · 기술(learnset/moves) · 아이템 · 진화체인 등 **부가 데이터**.

## 요구사항

### 기능 요구사항

- [ ] 18타입을 표현하는 타입 정의(예: `PokemonType` 유니온)와 타입 상성표를 단일 출처로
      제공한다. 상성은 공격타입→방어타입 한 방향으로 저장하고 배수는 0/0.5/1/2를 표현한다.
- [ ] 정답 후보 데이터는 후보마다 최소 `{ 폼 고유 식별자, 영문 식별자, speciesId(원종 연결),
      isDefault(원종 여부), formCategory(base|mega|primal|regional|other), 타입(1~2개),
      가능 특성(일반/숨은 구분 — **식별자/이름만**, 효과 메타데이터 없음), 세대 식별 정보 }`를
      포함한다.
- [ ] 이름 데이터는 후보마다 **한국어·영어 이름**을 제공한다. **폼 ko는 PokéAPI가 불완전**
      하므로 원종 ko + 폼 수식어로 합성한다(메가는 form_names ko 완성형 우선).
- [ ] 정답 후보 = **원종(전종 ~1025) + 타입 또는 특성이 원종과 다른 폼**(지역폼·메가진화·
      원시회귀)이다. 폼 포함 여부는 각 variety의 타입/특성 집합을 원종과 비교해 판정하며
      (이름 substring 매칭 금지), 거대화(gmax)·타입/특성 동일 폼·아르세우스·실버디는 제외한다.
- [ ] 외부 공개 소스(PokéAPI 등)에서 **빌드타임에 수집→정제(trim)→스키마 검증→정적 산출물
      생성**하는 스크립트를 둔다. 스크립트는 **외부 소스의 특정 커밋 SHA를 고정 참조**하고,
      그 SHA·제외 목록·블랙리스트를 산출물 메타데이터에 기록한다. 산출물은 저장소에 커밋되어
      런타임에 외부 네트워크 없이 로드된다.
- [ ] 도메인 코드용 조회 헬퍼를 제공한다(예: 후보 단건 조회, 두 타입 간 상성 배수 조회).

### 비기능 요구사항

- [ ] 18타입만 화이트리스트로 채택하고 비배틀 타입(stellar/unknown 등)은 제외한다.
- [ ] 데이터 산출물은 정적 번들로 제공하며, **gzip 합계 ≤ 50KB**(타입표+후보+이름)를
      예산으로 한다(notes 실측 시뮬레이션 ~18–26KB 근거; 폼 추가분 포함해도 여유).
- [ ] 스키마는 처음부터 **다세대 전제**(세대 식별 가능, 페어리 등 신규 타입 포함)로 설계해
      신세대 데이터가 추가돼도 깨지지 않는다. **세대 식별을 보장한다**(명시 저장 — ADR 확정).
- [ ] 계약은 TypeScript 타입으로 표현되고, 데이터가 그 타입을 만족한다(타입 안전성).
- [ ] 데이터 누락·모순(타입 0개, 특성 0개, 미정의 타입 참조, 빈 이름 등)이 있으면
      **검증/테스트가 실패**한다.
- [ ] **재수집 스크립트는 소스 스키마가 기대와 다르거나 필수 필드가 누락되면 비정상
      종료(빌드 실패)하고 부분 산출물을 커밋하지 않는다.**
- [ ] 데이터 출처와 라이선스(PokéAPI/api-data BSD-3, Showdown MIT, 스프라이트 CC0)와
      포켓몬 IP 고지를 표기한다(기존 README 고지와 정합).

## 수용 기준

- [ ] 타입 상성표가 알려진 케이스를 통과한다(vitest): 물→불=2, 전기→땅=0, 노말→고스트=0,
      페어리→드래곤=2, 강철→페어리=2, 드래곤→페어리=0.
- [ ] **18×18 상성표 전 셀이 권위 출처(PokéAPI `damage_relations`)에서 파생한 기대 스냅샷과
      전체 일치**한다(샘플 검증을 넘어 전체표 대조).
- [ ] 모든 정답 후보가 **1~2개 타입**, **1개 이상 가능 특성**, **세대 식별 정보**,
      **비어있지 않은 ko·en 이름**(폼은 합성 ko)을 가진다(스키마 검증 테스트).
- [ ] **정답 후보 수가 스냅샷 고정 기대값과 일치한다**: 원종 1025 − 블랙리스트(아르세우스·
      실버디) + 타입/특성 변경 폼(지역폼·메가·원시회귀 및 기타 intra-gen 변경 폼) = 약 1200여
      (~1180–1230, 테스트가 정확값을 하드코딩). 제외 목록·블랙리스트·사유를 산출물 메타데이터에
      기록한다.
- [ ] 후보 데이터가 `speciesId`로 같은 종의 여러 폼을 그룹핑할 수 있다(일일 정답의 동종 폼
      중복 출제를 막는 데이터 토대 — 선정 로직 자체는 비목표).
- [ ] **폼 필터 케이스 검증**(vitest): `vulpix-alola` 포함(타입 다름), `kyogre-primal` 포함
      (특성 다름), `charizard-gmax` 제외(원종과 동일), `deoxys` 폼 제외(동일), 아르세우스·
      실버디 전체 제외(블랙리스트).
- [ ] 조회 헬퍼가 알려진 입력에 기대값을 반환한다(예: 피카츄 전기 타입·정전기/피뢰침,
      ko "피카츄"; 알로라 나인테일 얼음/페어리).
- [ ] **데이터 산출물 gzip 합계 측정이 예산(≤50KB) 이내임을 빌드/테스트에서 검증**한다.
- [ ] 데이터 수집 산출물이 **결정론적**이다(고정 커밋 SHA 스냅샷 → 같은 산출물).
      빌드/런타임에 외부 네트워크 의존이 없다(산출물이 커밋되어 있다).
- [ ] `npm run harness:gate`(check/lint/build/test:run)가 통과한다.

## 열린 질문

- 특성·타입의 **한국어 표기**까지 수집할지(현재는 후보 이름 ko를 우선).
- 데이터 **갱신 주기/방법**(신세대 출시 시 고정 커밋 SHA를 올리는 재수집 PR 절차).
- 일일 정답에서 **같은 종의 여러 폼**(예: 리자몽·메가리자몽X·Y)이 같은 날 동시 출제되지
      않게 하는 그룹핑 — 본 계약은 후보를 모두 노출하고, 선정 정책은 정답 셀렉터 피처에서 결정.

## ADR 필요 여부

**필요 — `./adr.md`(proposed)에 7개 durable decision을 기록했다:** ① 데이터 수집 소스
(PokéAPI api-data 덤프) ② 생성 시점(빌드타임 정적 커밋) ③ 소스 리비전 pin(커밋 SHA)
④ 타입표 권위 출처(PokéAPI 1차·Showdown 2차) ⑤ 세대 표현(명시 저장) ⑥ ko 단일 종속·폼 ko
합성 ⑦ 정답 후보·폼 정책(타입/특성 변경 폼 포함, gmax·동일폼 제외, 아르세우스·실버디
블랙리스트) + 식별 단위((species, form) 복합키).

(정답 은닉 아키텍처는 본 피처 ADR이 아니라 후속 `guess-feedback-contract-adr` 피처에서 다룬다.)
