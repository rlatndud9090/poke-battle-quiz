# PRD: 퀴즈 데이터 계약

## 배경

이 프로젝트는 하루에 한 번, 정답 포켓몬을 배틀형 행동으로 추리하는 퀴즈를
제공한다. 유저의 하나의 행동은 하나의 턴으로 계산되고, 유저는 몇 턴 만에
정답을 맞혔는지 공유할 수 있어야 한다.

유저 행동은 자유 질문이 아니라 포켓몬 배틀처럼 보이는 명령이다. 정답 입력,
내가 기술 사용하기, 상대가 기술 사용하기의 결과가 타입 상성, 특성 발동,
기술 사용 가능 여부, 상태이상, 랭크 변화 같은 힌트를 드러낸다.

따라서 첫 구현은 전체 배틀 시뮬레이터나 전체 데이터 수집이 아니라, 퀴즈 진행에
필요한 데이터의 모양과 검증 가능한 최소 seed data를 정의해야 한다.

## 목표

- [ ] 유저 행동 1회가 턴 1 증가로 기록될 수 있는 데이터 기반을 마련한다.
- [ ] 정답 포켓몬 입력, 내가 기술 사용하기, 상대가 기술 사용하기에 필요한 데이터
      종류를 정의한다.
- [ ] 포켓몬, 타입, 기술, 특성, 상태이상, 랭크 변화 사이의 참조 무결성을 검증한다.
- [ ] MVP에서 검증 가능한 특성과 검증 불가능한 특성을 구분한다.
- [ ] 정답 후보는 검증 가능한 특성 선택지를 가진 포켓몬/특성 조합으로 제한할 수
      있게 한다.
- [ ] 추후 9세대 기준 전체 learnset 또는 generated data로 확장할 수 있는 경계를 둔다.

## 비목표

- [ ] 실제 포켓몬 배틀 시뮬레이터를 구현하지 않는다.
- [ ] 위력, 난수, 급소, 실제 명중률, 부가효과 확률을 계산하지 않는다.
- [ ] 전체 포켓몬, 전체 기술, 전체 특성을 처음부터 수록하지 않는다.
- [ ] 9세대 전체 learnset import pipeline을 이 작업에서 만들지 않는다.
- [ ] 회피율, 우선도 차단, 열매 연동처럼 아직 행동으로 검증하기 어려운 특성을
      억지로 정답 후보에 포함하지 않는다.
- [ ] UI 구현이나 공유 화면 구현은 이 작업에 포함하지 않는다.

## 요구사항

### 기능 요구사항

- [ ] 포켓몬 데이터는 id, 도감 번호, 한국어/영어 이름, 타입, 특성, 종족값,
      학습 가능 기술 참조를 표현해야 한다.
- [ ] 기술 데이터는 타입, 물리/특수/변화 분류, 대표 기술 역할, 상태이상 효과,
      랭크 변화 효과, 반동 같은 태그를 표현해야 한다.
- [ ] 유저가 사용하는 공격기는 각 타입별 물리/특수 대표 기술을 하나씩 제공할 수
      있어야 한다.
- [ ] 유저가 사용하는 변화기는 상태이상별 대표 기술과 상대 랭크를 변화시키는
      대표 기술을 제공할 수 있어야 한다.
- [ ] 기본 기술은 명중률 100으로 취급할 수 있어야 한다.
- [ ] 상대가 기술을 사용할 때, 학습 가능 여부와 특성에 의한 데미지 배율/타입
      변화/부가효과 변화를 표현할 수 있어야 한다.
- [ ] 특성 데이터는 발동 시점, 조건, 효과를 설명 가능한 구조로 표현해야 한다.
- [ ] 정답 후보 pool은 포켓몬 종만이 아니라 내부적으로 사용할 특성 선택지를
      제한할 수 있어야 한다.
- [ ] 검증 불가능한 특성은 밴 또는 보류 상태로 관리할 수 있어야 한다.

### 비기능 요구사항

- [ ] 데이터는 TypeScript 타입과 테스트로 참조 무결성을 검증할 수 있어야 한다.
- [ ] 초기 데이터는 공개 저장소에 안전하게 둘 수 있어야 한다.
- [ ] 외부 원천 전체를 그대로 저장하지 않고, MVP에 필요한 curated seed data로
      시작할 수 있어야 한다.
- [ ] 데이터 구조는 generated data가 나중에 추가되어도 domain API를 크게 흔들지
      않아야 한다.
- [ ] 검증 불가능한 특성을 숨은 예외로 두지 않고 명시적 상태로 남겨야 한다.

## 수용 기준

- [ ] `src/domain`에 퀴즈 데이터 계약 타입이 정의되어 있다.
- [ ] `src/data/curated`에 타입, 기술, 특성, 포켓몬, daily answer pool seed data가 있다.
- [ ] 각 타입별 물리/특수 공격 대표 기술이 하나씩 존재한다.
- [ ] curated 포켓몬이 참조하는 타입, 기술, 특성이 모두 존재하는지 테스트가 검증한다.
- [ ] daily answer pool이 검증 가능한 특성 선택지만 포함하는지 테스트가 검증한다.
- [ ] 검증 불가능한 특성이 데이터에서 `blocked` 또는 `planned` 상태로 구분된다.
- [ ] `npm run harness:gate`가 통과한다.

## 열린 질문

- 회피 관련 특성은 실제 확률을 적용할 것인가, 아니면 "회피할 뻔했다" 같은
  deterministic hint로 표시할 것인가?
- 정답 공유는 포켓몬 종만 공개할 것인가, 내부 active ability까지 공개할 것인가?
- 전체 9세대 learnset은 어느 시점에 generated data로 도입할 것인가?
- 폼 체인지, 리전폼, 특수폼은 MVP daily pool에서 언제부터 허용할 것인가?

## ADR 필요 여부

필요하다. 데이터 저장 형식, curated/generated 경계, 특성 trigger/effect 표현,
정답 후보 eligibility 정책은 구현을 제약하는 기술 결정이다.
